第零章節

OS的主要工作有:

  1. 分享電腦資源給多個不同的程式
  2. 管理並將底層硬體資源抽象化(提供介面)
  3. 共享硬體資源並使多個程式可同時運行
  4. 提供可控制的方式，使得程式之間可共享資料並合作

xv6效法Unix的設計，使用了傳統Kernel的形式(Kernel為一個特殊的程式，它將提供服務給正在運行中的程式)

如上圖，程式執行上有分為user space和kernel space，若proccess要使用kernel的服務則必須透過system call來呼叫

Process and Memory

當程式被載入到記憶體中運行時，稱之為Process。每個Process各自擁有自己的記憶體空間，並且擁有唯一的pid以供CPU辨別。
Process可以呼叫fork(一個system call)來建立一個child process，而這個child process的記憶體內容(而不是記憶體空間)與parent相同，並且fork在child和parent process都會return。
另外fork的回傳值在parent中為child的pid，而在child中回傳0。

// Example
int pid;
pid = fork();
if(pid > 0){
    printf("parent: child=%d\n", pid);
    pid = wait();
    printf("child %d is done\n", pid);
} else if(pid == 0){
    printf("child: exiting\n");
    exit();
} else {
    printf("fork error\n");
}

輸出:

parent: child=1234
child: exiting
parent: child 1234 is done

另一個重要的system call為exec，它可以用指定檔案的memory image覆蓋當前process的記憶體。換句話說當前process會執行指定檔案的內容。
而xv6 shell的做法就只是透過fork建立child process，之後child process會call exec來執行用戶指定的程式，程式邏輯非常簡單明瞭。

I/O and File descriptors

File descriptor是一個整數，它可以代表一個文件、設備或者是Pipe的抽象。
在xv6之中，可以用read和write對一個file descriptor所描述的對象做讀取或寫入，也因為file descriptor的抽象特性，程式本身不必得知它輸入輸出的對象是誰。

每個process各自擁有一個process table，而file descriptor則是table的index(從0開始)。
一般來說，0代表standard input，1代表standard output，2代表standard error。
shell就利用了這個standard input/output的慣例來實現I/O redirection。

這裡透過cat的實作可以明白I/O redirection

char buf[512];
int n;

for(;;){
	n = read(0, buf, sizeof buf);
	if(n == 0)
    	break;
    if(n < 0){
        fprintf(2, "read error\n");
		exit();
	}
    if(write(1, buf, n) != n){
    	fprintf(2, "write error\n");
        exit();
	}
}

下面則是shell 執行 cat < input.txt的程式瑪

char *argv[2];
argv[0] = "cat";
### argv[1] = 0;
if(fork() == 0) {
	close(0); // 確保下一行一定會建立在0上
	open("input.txt", O_RDONLY);
    exec("cat", argv);
}

可以看到cat的輸入輸出是直接從standard I/O上進行，而shell在呼叫cat前call了close(0)來確保0已經關閉了，所以下一行的open會直接使用0做為file descriptor(會從最小的值建立)。

另外補充child process和parent process會共享offset
因此

if (fork() == 0) {
    write(1, "hello ", 6);
    exit();
} else {
    wait();
    write(1, "world\n", 6);
}

會在standard output輸出

hello world

Pipe

Pipe 是一個kernel buffer，它提供process一個可以將兩個file descriptor對象做讀取與寫入的服務。

這裡透過wc的範例來解釋pipe的用途

int p[2];
char *argv[2];
argv[0] = "wc";
argv[1] = 0;
pipe(p);
if(fork() == 0) {
	close(0); // 確保下一行一定會建立在0上
	dup(p[0]); // 複製一份p[0]的file descriptor
	close(p[0]);
	close(p[1]);
	exec("/bin/wc", argv); // 執行計算字數的程式
} else {
	write(p[1], "hello world\n", 12); // 將hello world寫入p[1]
	close(p[0]);
	close(p[1]);
}

這個Process先fork出一個child，之後parent process會write “hello world\n"到p[1]，而child process會將p[0]複製到standard input，讓wc去計算字數。
透過pipe的效果，p[0]會讀取到"hello world\n”，進而讓wc去計算它的字數。

File system

xv6的File system是一個樹狀結構，從root開始延伸擴展。
表示方法: /a/b/c代表根目錄root之下的a目錄下的b目錄下的文件c。
不從/開始的幕路則是當前proces目錄下的相對位置。

有幾個常用於file system的system call:
chdir 用於改變當前process的所在目錄
mkdir 建立新目錄
mknod 建立新的設備文件
fstat 取得文件狀態
link 連結文件與文件名
unlink 移除連結

fstat取得資料的struct如下定義:

#define T_DIR 1 // Directory
#define T_FILE 2 // File
#define T_DEV 3 // Device
struct stat {
    short type; // Type of file
    int dev; // File system’s disk device
    uint ino; // Inode number
    short nlink; // Number of links to file
    uint size; // Size of file in bytes
};

值得一提的是ino這個參數，它代表這份文件的文件名有幾種。任一個文件都可以透過link來建立不同的文件名稱，當ino為0時才表示該文件已完全清除。

第一章節

Kernel 可分為兩種類型:

1. monolithic kernel

OS所有程式都運行在kernel mode下，擁有完整的hardware privilege。設計上簡單且效能高，但是若發生錯誤會導致crash。

2. microkernel

為了降低發生錯誤的風險，盡可能地減少kernel mode下的程式數量，大部分的程式執行於user mode，讓kernel盡可能輕量化。

Process

xv6使用page table為每個Process提供一個專有的address space，page table則將virtual address轉換成physical address

如上圖，每個process都有一個這樣的address space。它從virtual address的0位置開始依序存放程式指令、global variables、stack、heap(for malloc)。

每個Process的執行單位為Thread，系統在切換process時其實就是暫停一個thread並運行另一個thread。而每一個Thread都擁有自己的stack，用於存放Thread的大多數狀態(local variable, return address)。

每個Process都有user stack和kenrel stack，當process執行一般用戶指令時，只會用到user stack。當呼叫system call時，便會使用kerenl stack，其user stack仍會保留原本的data。

The first address space

電腦開機時，會從硬碟載入boot loader到memory並執行。
boot loader則會將xv6的kernel從硬碟中載入至0x100000並從entry執行。


entry會建立一個page table，並將0x80000000(KERNBASE)開始的virtual address map到physical address 0x0，最後是將KERNBASE+0x400000 map到0x400000，它將限制kernel code必須在4MB內。

entry在執行C的main以前，它先將%esp指向stack的address，然後用indirect jump到main。

Creating The first process

main會Call userinit建立第一個process。而userinit會先call allocproc，它會在process talbe中尋找一個UNUSED的slot，找到後將其改為EMBRYO後分配給該process並給它一個唯一的pid。

user mode到kernel mode是通過interrupt實現，每當要切換到kernel mode時，xv6的trap entry code會保存user resgiters至process’s kernel stack。userinit會把中斷點寫至新的stack頂部，讓kernel執行完成後可以回到中斷的程式碼繼續運行。

初始化完成後userinit會將p->state改為RUNNABLE。

Running the first process

userinit完成後，mpmain會call scheduler來安排process運行。
scheduler會尋找RUNNABLE的process(這時候只有initproc)，之後用switchuvm讓硬體使用page table。

initcode.S會call exec，執行流程如下

# exec(init, argv)
.globl start
start:
  pushl $argv
  pushl $init
  pushl $0  // where caller pc would be
  movl $SYS_exec, %eax
  int $T_SYSCALL # int是指interrupt，這行表示執行system call(根據上一行，這個system call為exec)

# for(;;) exit();
exit:
  movl $SYS_exit, %eax
  int $T_SYSCALL
  jmp exit

# char init[] = "/init\0";
init: 
  .string "/init\0" # exec會執行/init

# char *argv[] = { init, 0 };
.p2align 2
argv:
  .long init
  .long 0

/init會反覆執行shell、處理zombie process(沒有parent的process)，直到離開shell，系統就能運行完成了。

項目三

第三章節 說明運作原理、程式流程及資料結構（註明哪一個檔案內的functions、程式裡面所使用到哪些變數及結構）

System calls, exceptions and interrupts

CPU在執行process時，會不斷地做這三件事:

  1. Read instruction
  2. Advance the program counter
  3. Execute the instruction

而有幾種情況下會需要用到kernel服務，而不是繼續執行user instruction。這時候就需要用到x86中很重要的概念Interrupt。
一般來說有這幾種情形:

  • System call: 使用者程式需要用到kernel服務時會使用
  • Exceptions: 當程式進行非法操作時會觸發它來中斷程式
  • Interrupts: 硬體產生希望引起OS注意的訊息，像是clock chip每100毫秒會產生一次interrupt讓kernel可以實現time sharing

xv6用了trap來表示一個由cpu上運行的process所導致的中斷，而interrupt是由設備所導致的。
trap和interrupt只有一個1bit不同，它是eflag中的IF bit。
因此在trap正在執行時，interrupt還是可以去中斷它來回應硬體訊息。

X86 protection

x86有4個保護層級，數字0(最高權限)到3(最低權限)。大部分的OS都指使用兩個權限0和3，分別做為kenrel mode和user mode。當前process的指令權限存放在%cs的CPL欄位。

x86透過Interrupt Descriptor Table(IDT)來定義interrupt process的入口(entry)，而IDT有256個入口，每一個都提供了對應的%cs和%eip。

一個程式要在x86上進行system call時，它需要call int n指令，n表示IDT的index，而int會進行下面的步驟:

  • 從IDT中取得index為n的descriptor。
  • 檢查%cs的CPL <= DPL，DPL代表descriptor的權限層級。
  • 若Target segment selector的 PL < CPL，則保存%esp和%ss至CPU內部暫存器。(發生權限轉換)
  • 從descriptor中載入%ss和%esp
  • Push %ss
  • Push %esp
  • Push %cs
  • Push eip
  • 若是Interrupt(由硬體呼叫的)，清除eflag中的IF bit
  • 將%cs和%eip設定為descriptor中的值


上圖是發生了權限轉換後的kernel stack狀態。

OS可以用iret來從一個int指令中返回。

int指令會觸發中斷，而x86定義了256種中斷，0-31定義為software exceptions(如除0錯誤)，32-63定義為硬體中斷，64做為system call的中斷。

C中斷process

當process觸發trap時，會執行以下程式碼:

void
trap(struct trapframe *tf)
{
  if(tf->trapno == T_SYSCALL){ // system call中斷
    if(myproc()->killed)
      exit();
    myproc()->tf = tf;
    syscall(); // 就去call systemcall() function
    if(myproc()->killed)
      exit();
    return;
  }

  switch(tf->trapno){ // 硬體中斷
  case T_IRQ0 + IRQ_TIMER:
    if(cpuid() == 0){
      acquire(&tickslock);
      ticks++;
      wakeup(&ticks);
      release(&tickslock);
    }
    lapiceoi();
    break;
  case T_IRQ0 + IRQ_IDE:
    ideintr();
    lapiceoi();
    break;
  case T_IRQ0 + IRQ_IDE+1:
    // Bochs generates spurious IDE1 interrupts.
    break;
  case T_IRQ0 + IRQ_KBD:
    kbdintr();
    lapiceoi();
    break;
  case T_IRQ0 + IRQ_COM1:
    uartintr();
    lapiceoi();
    break;
  case T_IRQ0 + 7:
  case T_IRQ0 + IRQ_SPURIOUS:
    cprintf("cpu%d: spurious interrupt at %x:%x\n",
            cpuid(), tf->cs, tf->eip);
    lapiceoi();
    break;

  //PAGEBREAK: 13
  default: // software exceptions的中斷
    if(myproc() == 0 || (tf->cs&3) == 0){
      // In kernel, it must be our mistake.
      cprintf("unexpected trap %d from cpu %d eip %x (cr2=0x%x)\n",
              tf->trapno, cpuid(), tf->eip, rcr2());
      panic("trap");
    }
    // In user space, assume process misbehaved.
    cprintf("pid %d %s: trap %d err %d on cpu %d " // 輸出錯誤訊息
            "eip 0x%x addr 0x%x--kill proc\n",
            myproc()->pid, myproc()->name, tf->trapno,
            tf->err, cpuid(), tf->eip, rcr2());
    myproc()->killed = 1;
  }

  // Force process exit if it has been killed and is in user space.
  // (If it is still executing in the kernel, let it keep running
  // until it gets to the regular system call return.)
  if(myproc() && myproc()->killed && (tf->cs&3) == DPL_USER)
    exit();

  // Force process to give up CPU on clock tick.
  // If interrupts were on while locks held, would need to check nlock.
  if(myproc() && myproc()->state == RUNNING &&
     tf->trapno == T_IRQ0+IRQ_TIMER)
    yield();

  // Check if the process has been killed since we yielded
  if(myproc() && myproc()->killed && (tf->cs&3) == DPL_USER)
    exit();
}

System calls

system call可以透過argint、argptr、argstr 和argfd分別取得int、pointer、string和file descripot類型的參數。其中%esp會指向system call的return address。%esp指向system call結束後的return address，而參數就在%esp+4的位置，因此第n個參數的位置就等於%esp+4+4*n。

Interrupts

電腦上的設備/硬體可以產生Interrupt，而xv6需要配合硬體來處理中斷訊號。
Interrupt與system call相似，不過它可以在任何時候對CPU發送信號。像是使用者用鍵盤輸入時，硬體會發送一個Interrupt到CPU，令CPU去執行相應的動作。

CPU也可以透過改變%eflags暫存器中的IF欄位(1 bit)來接收中斷訊號，cli可以清空IF來關閉(無視)中斷訊號；而sti 則可以開啟接收中斷訊號。xv6在啟動cpu時會關閉中斷訊號，而在scheduler啟動時會開啟中斷訊號，另外在一些指令(其實就是硬體Interrupt類型的指令)執行中也會關閉中斷訊號。

Drivers

Driver是OS中用於管理指定設備/裝置的程式碼，它提供設備相關的中斷處理程序，能控制設備完成工作也能產生中斷訊號給CPU等等，它也必須熟悉設備的I/O接口，因此Driver在設計上可能會非常難以撰寫。

Disk driver

電腦開機時，main會call ideinit來初始化disk driver，它會透過picenable和ioapicenable來打開IRQ_IDE的中斷。

void
ideinit(void)
{
  int i;

  initlock(&idelock, "ide");
  ioapicenable(IRQ_IDE, ncpu - 1);
  idewait(0);

  // Check if disk 1 is present
  outb(0x1f6, 0xe0 | (1<<4));
  for(i=0; i<1000; i++){
    if(inb(0x1f7) != 0){
      havedisk1 = 1;
      break;
    }
  }

  // Switch back to disk 0.
  outb(0x1f6, 0xe0 | (0<<4));
}

之後 ideinit會檢查硬碟，它用idewait來等待disk接受指令，而PC motherboard透過I/O port 0x1f7來表示硬碟的status bits。直到busy bit(IDE_BSY)被清除以及ready bit(IDE_DRDY)被設定時才會讀取status bits。

ideinit完成後，在buffer cache使用iderw以前，不會再使用硬碟。

if(!holdingsleep(&b->lock))
panic("iderw: buf not locked");
if((b->flags & (B_VALID|B_DIRTY)) == B_VALID)
panic("iderw: nothing to do");
if(b->dev != 0 && !havedisk1)
panic("iderw: ide disk 1 not present");

iderw可以根據flags來刷新一個鎖住的buffer，若B_DIRTY為1則將buffer cache的內容寫進硬碟，若B_VALID為0則將硬碟資料寫到buffer cache裡。

由於硬碟的讀取時間對於CPU來說太慢了，因此OS會讓CPU在等待時間去做其它事情，而硬碟操作完成時送出一個中斷給CPU。iderw會維護一個queue of requests，讓硬碟控制器每次處理一個操作。

如下，iderw會將b(buffer)寫入queue的末端。

// Append b to idequeue.
b->qnext = 0;
for(pp=&idequeue; *pp; pp=&(*pp)->qnext)  //DOC:insert-queue
;
*pp = b;

如下，通過idestart來執行queue上的硬碟操作

// Start disk if necessary.
if(idequeue == b)
idestart(b);

如下，當iderw會執行sleep直到硬碟操作完成。這個時候會讓CPU進行其它process來維持工作狀態。

// Wait for request to finish.
while((b->flags & (B_VALID|B_DIRTY)) != B_VALID){
sleep(b, &idelock);
}

最後，硬碟完成自己的操作後觸發一個中斷。trap會用ideintr來處理它。

// Interrupt handler.
void
ideintr(void)
{
  struct buf *b;

  // First queued buffer is the active request.
  acquire(&idelock);

  if((b = idequeue) == 0){
    release(&idelock);
    return;
  }
  idequeue = b->qnext;

  // Read data if needed.
  if(!(b->flags & B_DIRTY) && idewait(1) >= 0)
    insl(0x1f0, b->data, BSIZE/4);

  // Wake process waiting for this buf.
  b->flags |= B_VALID;
  b->flags &= ~B_DIRTY;
  wakeup(b);

  // Start disk on next buf in queue.
  if(idequeue != 0)
    idestart(idequeue);

  release(&idelock);
}

ideintr會查詢第一個buffer是否在進行操作，若它正被讀取且硬碟控制器有資料在等待中，它就會調用insl將資料載入buffer。
buffer準備完成後，ideintr會把B_VAILD設為1且B_DIRTY設為0，之後wakeup buffer內的process。
最後ideintr會把下一個等待中的buffer繼續執行idestart。
