項目二

作業系統的綱領

  1. 將電腦的效能在有限的硬體資源上提供有用的服務給多工的工作
  2. 創造良好的互動介面，使執行時非常方便
  3. 提供複雜的程式但設計的很簡單

Process

  1. 每份執行的程式program稱作process
  2. 當process需要實行核心kernel服務時，它會實行procedure call在作業系統的介面
    • 稱作「system call」，kernel完成運算後將資料回傳
  3. 因此process在user space, kernel space中輪流執行

Xv6內核

Xv6採用了傳統內核的概念，User經由shell與cat下達指令，將程序(process)傳至kernel進行運算，包含指令、數據、內存空間的資訊。

Xv6擁有使用者（指令、數據、stack）、內核專用的memory空間，Xv6提供了時間分工的特性，分配CPU與暫存器的互動，例如Process碰上中斷或等待中時能由Xv6保存資料直到下次再度執行

Xv6 System Call摘要

以下是System的基本System call介紹：

fork：

fork可創建子進程，執行過程中等到子進程完成後，父進程的wait()將return並繼續父進程接下來的程式碼，改變其中一邊的變數並不影響

exit:

可終止process，釋放memory

exec:
可將舊有的資料用來自file system的新資料取代，此舉需兩個參數：包含可執行文件的file名稱、一組字串

Xv6 shell:

總和以上

  1. 主循環通常透過指令getcmd讀取輸入的指令
  2. 生成fork的副本
  3. 指令runcmd執行
  4. 若指令中需要exec則子process將執行exec的目標
  5. 結束後exit將返回父process的wait()

如此便能將上述System call做一個完整的應用

File descriptor

File descriptor 是個整數，他代表了process可以被內核管理的對象，每個Process都有張表做參考

Pipe

Pipe身為小的內核kernel緩衝區，提供Process間的溝通互動

文件系統

Xv6文件系統提供文件(未翻譯的陣列)及目錄(指向文件或其他目錄的地址)

Process 概覽

  1. 提供記憶體空間、供人搜尋的地址，並將虛擬地址(X86使用)翻譯(map)為物理地址(處理器chip傳送給main memory)，從地址最低處開始放置

  2. 在記憶體中儲存內核的指令與數據，將其放置於0x80100000開始

  3. 使用struct proc來維護process的狀態，保護最重要的page table, kernel stack以及運行狀態

  4. Process底下的thread成了執行process的指令，能夠停止等待、轉移執行任務

  5. p->state指示process的狀態

  6. p->pgdir持有page table

PC從開機乃至初始化第一個Process

  1. 初始化並載入boot loader至記憶體並運行

  2. 從disk中載入entry

  3. 將entrypgdir的物理位址載入控制register的%cr3中

  4. entry轉至kernel的C語言，並在高地址記憶體中執行，首先將stack pointer指針%esp指向一段被用來當作stack的memory

  5. Kernel在高記憶體位址中執行function main

  6. Main初始化設備與子系統，透過呼叫userinit建立第一個process

    • 首先呼叫allocproc (在page table中分配一個slot並初始化process的狀態）
    • userinit僅被呼叫一次，allocproc 每次建立process時都會呼叫

  7. allocproc在proc中找到標記UNUSED的slot，因為被使用則標記為EMBRYO並提供pid

  8. allocproc嘗試分配kernel stack給kernel中的threads，失敗則將slot狀態寫入UNSED並return 0標記失敗

  9. allocoproc準備特別的kernel stack, registers，設定第一次執行時將return至使用者空間

  10. kernel’s thread從p->context->eip指向forkret，讓thread從forkret開始執行

  11. 執行initcode.S

  12. 定義_binary_initcode_start和_binary_initcode_size

  13. userinit呼叫inituvm，接著把trap frame設為初始用戶模式

  14. 初始化完成後userinit將p->state設為RUNNABLE使process能被呼叫

進行第一個Process

  1. 在main呼叫userinit，mpmain呼叫scheduler開始執行process

  2. scheduler找到p->state為RUNNABLE的process:initproc

  3. 呼叫switchuvm通知硬體開始目標process的table

  4. scheduler將process的p->state設為RUNNING並呼叫swtch

  5. swtch保存當前register並將目標kernel’s thread仔入x86硬體register

  6. 指令ret pop目標process的%eip，結束context的交換，如此processor就在process p 的kernel stack上執行

第一個system call:exec

  1. 觸發exec，用新的process代替當前program的memory與register

    • file descriptors, process id 和parent process不變

  2. initcode.S將$argv, $init $0push進stack中，並將%eax設為SYS_exec，接著執行 int T_SYSCALL

以上的兩項操作目的為告訴kernel執行exec這system call
操作順利的話exec將不會return，接著執行$init

大多數的操作系統皆採用了process的概念，且與XV6十分相似

但真正的OS會利用明顯的free list在constant time內尋找空閒的proc，而不向alloproc花費linear-time

Xv6為了簡單採納linear-search，找第一個空閒的proc，但Xv6的地址空間無法使用超過2GB的物理RAM
->最好的解決方法為使用64-bit的機器

Chapter 4 Locking

摘要

Locks的主要功能為保證每次使用檔案時只有一個CPU會使用，在這情況下所可以保護資料

struct list {
int data;
struct list *next;
};

struct list *list = 0;

void
insert(int data)
{
struct list *l;

 { = malloc(sizeof *l);
l->data = data;
l->next = list;
list = l;
 }
}

因此，若在insert時加入輸出語句改變執行時間便可使競爭消失

競爭關係(Race Condition)

同一個記憶體位置同時被訪問，至少有個訪問是被複寫(write)的
通常此種情況是個bug，將失去資料更新、獲得不完整的資料

解決方案
->1.兩個CPU的確切時間
->2.記憶體如何安排記憶系統，使錯誤減少
->3.使用鎖

Lock保證一次只有一個CPU可執行(稱作互斥mutual exclusion)

鎖保護一些適用於數據的不變量
(Invariant：數據結構在操作中維護的一些狀態)，以上述程式碼的案例中list即為不變量

操作的正確行為往往取決於不變量是否為真
雖然操作中有可能暫時破壞不變量，但結束操作前須復原

鎖

Xv6有兩種鎖：spin-locks, sleep-locks

struct list *list = 0;
struct lock listlock;

void insert(int data)
{
    struct list *l;
       acquire(&listlock);
    l = malloc(sizeof *l);
    l->data = data;
    l->next = list;
    list = l;
       release(&listlock);
}

void acquire(struct spinlock *lk)
{
24    for(;;) {
25        if(!lk->locked) {
26            lk->locked = 1;
27           break;
        }
    }
}

----程式碼講解----

  • Spin-Lock採用結構體(struct spinlock)，當鎖可以被獲得時locked = 0

  • 但有可能有兩個以上的CPU同時執行第25行，發現lk->locked為0，因此而拿到了鎖，違反了互斥的現象

  • 因此，lk->locked = 1(26行)與break(27行)必須不可分割地同時執行(atomic step

xchg的運用

xchg屬於Xv6中的特殊指令，執行時將記憶體中的word與register內容交換

Acquire透過鎖的幫忙後：

  • 可獲得鎖的CPU和stack訊息
  • 當鎖被忘了釋放或debug時可幫助找到問題
  • 唯有在持有鎖時才能修改

Release：

  • 執行Acquire的相反：釋放鎖
  • 採用集合式指令清除lock，xchg才不會阻擋更新locked
  • movl可幫忙不可分割的更新

Acquire vs Release: critical section

鎖的運用

該決定該使用幾個Lock、哪些數據與不變量該被保護成了使用鎖的關鍵

  1. 當一變數可被依CPU撰寫時，同一時間第二個CPU亦可讀或寫，Lock要保護避免兩個執行重疊
  2. 若不變量涉入多個記憶體空間，他們全都該被單獨一個鎖保護來維持不變量

giant kernel lock
當效率問題不在考量內時：

  • 可不必考慮鎖的數量
  • 進入內核kernel上鎖，離開時解鎖
  • 一段單位時間中只有一個CPU能被運行
    • 若想依靠kernel進行大量的運算，需使用更精細的鎖來讓CPU可同時輪流運行

給每個Process table中的Thread都上鎖
->如此一來Tread就能並行執行了，相反的鎖愈多情況愈複雜

死鎖與鎖的順序(Deadlock and lock ordering)

若一段code需使用複數個鎖，則順序必須遵守，否則將發生死鎖deadLock

  3. 按照以上順序解鎖

中斷處理程序Interrupt handlers

即便是單一處理器中斷亦可造成併發

----中斷程序案例1----

假設中斷處理程序interrupt handler使內核code暫停，指令iderw在持有idelock的情況下將被指令ideintr中斷

但ideintr需要idelock<->ideintr執行完成才能回到指令iderw並釋放鎖，
->此舉將造成deadlock

為避免以上情況，當中斷處理程序需要鎖(spin-lock)時，processors處理器就不能在允許中斷發生時擁有鎖

----中斷程序案例2----

  1. 產生中斷的處理程式在持有鎖的狀態下發生中斷
  2. 下一個CPU提出要求鎖的指令
  3. 但程式已經中斷鎖尚未釋放->導致要求鎖的CPU被拒絕請求

->導致死鎖

----Xv6案例----

interrputs的acquire必須等待其他處理器的thread釋放鎖

----屏除中斷----

  • Acquire 在獲得鎖之前用 pushcli
  • Release 在解除鎖之後用 popclil屏蔽中斷

Instruction and Memory Ordering

假設程式按照一定的規則處理code、安排鎖

編譯器與處理器往往為了高效能而打破規則

  • 若指令需要多個週期，處理器希望指令能盡早開始，其他指令便能並行執行避免耽誤
  • 例如程序B較快，先做B再做A
  • 有可能讓軟體暴露於危險之中，導致不正確的結果

->Xv6在acquire, release採用_sync_synchronize()避免編譯器與CPU reorder(重新排序)讀檔與存檔

Sleep Locks

將常數寫入磁碟時將耗費許多時間，其他處理器可善用這段時間處理其他事

Xv6提供堪用的lock使交換context變得容易->sleep locks

Sleep Lock特性

  1. Sleep-lock擁有Locked Field且具Spin-Lock的保護

  2. Acquiresleep呼叫sleep時可atomically接受CPU的調度並釋放Spin-Lock，其他threads可在Acquiresleep等待時執行

  3. Sleep-lock因留下允許中斷點，則其不可成為中斷點處理者

  4. Spin-lock可用於Sleep-lock中，但Sleep-lock不可用於Spin-lock

  5. 多數時間Xv6採用Spin-lock

    • File System才採用Sleep-lock，如此才能維持有如硬碟程序長度般的鎖

Limitations of Locks

Locks通常能解決「同時發生」的問題

Function 有可能被擁有鎖的程式碼與不需要擁有鎖的程式碼

因此function需要兩個變數：

  1. 獲得鎖
  2. 期待呼叫者已經持有鎖

現實情況Real World

  1. 多數作業系統採用atomic instructions

    • 支援並行線程pthread(POSIX thread)
    • 允許使用者同時在不同處理器上執行複數個Thread

  2. 當一個pthread被System Call阻擋時，同樣Process中別的pthread應該要能夠在同一個處理器上被執行

  3. 若pthread改變了process的地址空間，Kernel必須通知其他Processor並更新地址資訊

  4. 當A處理器的鎖被B處理器所需要時，atomic instruction將從A的快取line中找到鎖並轉移至B快取Line

    • 此舉有可能讓這行緩存的其他備份失效
    • 從處理器的快取中提領將比Local快取還要來的耗費效能

  5. 基於使用鎖的困擾與昂貴的代價，許多作業系統採用避免Atomic操作或使用鎖

    • 例如採用Linked-List取代
