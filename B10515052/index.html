撰寫第0、1章節的報告

第0章在描述OS的運作方式以及system calls，作業系統的工作是將電腦的資源分配給許多的程式，並管理記憶體、以及檔案，而使用者從user space下system call指令來控制Kernel

第1章在描述OS的組織架構，OS最重要的功能就是讓許多程式能夠同時運作而不會產生Error，並有效地分配CPU的使用時間給程式，因此記憶體的資料管理就很重要，否則程式雖然沒有Error，執行出來結果卻是錯的。

項目三

Chapter 4: Locking

在作業系統中不論是單核心或是多核心CPU，Lock都非常重要，以確保多工處理時資料不會出錯。

Race Condition

首先要介紹的是Race Condition，其代表同時間在某個記憶體位址有多個指令在寫入資料進入該記憶體位址，或是讀取尚未更新的資料，為了避免race發生，最常使用的方式就是lock，以下程式碼以及流程圖指示出執行insert()時的方式以及流程(備註部分是加上Lock所需的code):



















struct list {
    int data;
    struct list *next;
    struct lock listlock;
    };
    
struct list *list = 0;

void
insert(int data){
    struct list *l;
    //acquire(&listlock);
    l  = malloc(sizeof *l);
    l->data = data;
    l->next = list;
    list = l;
    //release(&listlock);
    }

但是此方法仍有缺點，就是當兩個CPU幾乎同時執行acquire()時，仍然無法避免Race Condition，然而，在xv6中他利用了xchg的指令，xchg會將記憶體與暫存器中1個word大小的內容交換，而acquire()會不斷執行xchg，如此當CPU1 lock 記憶體的時候，CPU2便無法再lock記憶體。

Deadlock

所謂的Deadlock是指程式在acquire時互相要求對方已經lock住的memory，導致產生互不相讓的情形，就稱為Deadlock，需要注意的是，即使目前是single processor，Deadlock依然會發生。

舉例來說，當目前processor是可以被中斷的，且kernal可能隨時被中斷去執行interrupt handler，假設iderw hold idelock，然後被中斷去執行ideintr，而ideintr嘗試lock idelock，發現已經被hold後便一直等到它被release，但idelock現在只能被iderw給release，但如果iderw要繼續執行並release idelock就必須等到ideintr return，如此不只是這個processor，連整個系統都在Deadlock的狀態裡。

為了解決這個問題，OS(以xv6為例)在processor lock memory source的時候，會把interrupt給disable，此時interrupt仍會發生在其他processor上，而其他processor上的interrupt仍可等到該processor release memory source，此方法便可解決同一processor上因interrupt而產生的Deadlock。

Instruction and memory ordering

雖然我們通常覺得程式在執行時，會依照程式碼的順序一步一步執行，其實不然，為了增加效率，complier跟processor可能會讓需要執行較多cycles的指令優先執行，讓他能和其他指令同時執行，讓CPU不會處在沒工作的情況，但這樣的reordering很有可能會導致錯誤，以下程式碼為insert()範例:







acquire(&listlock);
l  = malloc(sizeof *l);
l->data = data;
l->next = list;
list = l;
release(&listlock);

當compiler對第3~5行的指令與第6行進行reorder的話就會發生很嚴重的錯誤，為了告訴硬體以及compiler不要reorder，在xv6在acquire 與 release中使用叫做 __sync_synchronize()的記憶體保護方式，在xv6/spinlock.c中被使用，以下為acquire以及release的程式碼:

Sleep Lock

xv6有時必須hold lock很久，像是在進行file system的操作時可能需要等上幾十毫秒，這對processor算是很久的，此時就必須讓processor跳出去執行其他thread(執行緒)，這樣執行起來才有效率，而不是讓processor在那裡乾等，xv6就使用了sleep-lock型態的lock。

而sleep-lock的設計有一項問題需要去克服: 假設現在T1 lock了 L1，然後yeild之後T2 想要 lock L1，此時就必須保證 T1 還能夠繼續執行，這樣 L1 才能夠被release，但如果此時interrupt被disable了，那麼 T1 就無法繼續執行下去，此時就產生了deadlock，為了避免deadlock產生，xv6使用了acquiresleep並且保持interrupt enable，也因此sleep-lock不能在interrupt handler中執行，而xv6也只會在file system中使用sleep-lock。
