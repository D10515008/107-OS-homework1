eax 可用來做加減乘除運算、邏輯運算、字串運算，或是I/O處理。
ecx 可用來進行資料運算、存放迴圈次數、字串處理的重複次數、資料位移或是旋轉的次數。
edx 可用來進行資料運算、乘除運算，或是拿來存放I/O的位址。
ebx 可用來進行資料運算，或是加強索引及定址功能。可以直接當作記憶體位址的基底運算元。
esp SP永遠指向堆疊頂端的最新資料儲存位址。當堆疊資料有進(PUSH)出(PUP)時，SP的位址也會隨之加減更動。
eip 用來指向指令的所在位置，負責掌管CPU執行程式的流程。
eflags 為IF(interrupt flag)，當IF=1時，CPU可以接受外部的中斷；IF=0時，則無法接受外界中斷。

項目二：

chapter 0:

作業系統是軟體與硬體之間的橋樑，當軟體需要用到電腦的硬體資源如(I/O, 記憶體)時，軟體可以透過作業系統的 System call 來完成，這樣的設計一方面可以保護硬體不被軟體不當的使用，也可以適當的分配硬體資源給每個 process 使用。
Xv6 可以擁有多個process因為它可以time_share process，當一個process要新增一個新的process時，他會執行fork指令產生出一個新的process，新的process跟舊的process會有相同的memory content，但是如果新的process改變自己的變數，舊的process的變數不會被改變，反之亦然。
file descriptor是使用正整數來代表kernal 所管理的物件，0是讀入的地方(standard input)，1是輸出的地方(standard output)，2是錯誤訊息寫入的地方(standard error)，file descriptor要新增時，會選用最小的正整數做為新的file descriptor。不管新增的是file, device或pipe，file descriptor都是一個正整數。

Pipes 是一對file descriptors，pipe的一端用來寫入資料，另一端用來讀取資料，pipe提供了一種方法讓process見彼此可以溝通。
Xv6的File system從root開始，使用’/’就可以進入資料夾。

chapter 1:

作業系統需要符合三個條件，multiplexing(同時執行多個process)，isolation(不會因為某個process有bug而影響到其他process)，interaction(process之間可以互相溝通，比如用pipe)。
X86的processor有kernal mode與user mode，在kernal mode時，process
被允許使用privileged instructions(如reading memory，I/O)，但是如果有application在user mode時去使用privileged instructions，process就會進入kernal mode，然後把這個application清掉。
Xv6會用page table給每個 process 一個專屬的memory空間(virtual address)，
之後要執行的時候再把virtual address轉成physical address。
Xv6把process的狀態儲存在 struct proc裡面，每個process都有user stack(執行user的code)與kernal stack(執行kernal 的code)，這樣的設計可以確保當user stack壞掉時，kernal還是能夠正常執行。

第一個執行的process:

Main 會呼叫userinit，userinit會呼叫allocproc，allocproc會新增一個struct proc再page table裡，然後初始化process state讓kernal的thread可以執行。
Mpmain 會呼叫scheduler，scheduler會找到initproc上面的p->state顯示runable然後把他設定成p->state running，最後ret instruction pop %eip from the stack，然後processor開始跑在kernal stack上的p程式。

項目三：

Chaper 4 lock:

為什麼要有Lock?

電腦在執行時，可能會發生有兩個CPU同時想要使用某段記憶體的情況，如果這種情況發生，就會導致記憶體儲存到不正確的值，甚至是破壞那段記憶體的資料結構。

race condition


















struct list {
int data;
struct list *next;
};

struct list *list = 0;

void
insert(int data)
{
struct list *l;

l = malloc(sizeof *l);
l->data = data;
l->next = list;
list = l;
}

以上是一段Link_list的程式碼，如果段程式碼每次都只有一個process在執行，是不會有問題的，但是如果有兩個process同時執行這段程式碼，而且他們都剛好執行到第十五行的l->next = list，那當他們執行第十六行的時候，第一個list的assignment就會被第二個assignment蓋過去，這種情況叫做 race condition。
要解決這個問題，就可以使用Lock。我們只要在上面那段程式碼加上acquire(&listlock)跟release(&listlock)就可以避免兩個process同時執行這一段程式了。
















struct list *list = 0;
struct lock listlock;

void
insert(int data)
{
struct list *l;

acquire(&listlock);
l = malloc(sizeof *l);
l->data = data;
l->next = list;
list = l;
release(&listlock);
}

Xv6有兩種lock，一種是spin lock，另一種是sleep lock，首先先介紹spin lock，
Spin lock的資料結構如下:

Spin lock:










struct spinlock {
uint locked; // Is the lock held?(0是沒有被占有，1是被占有)

// For debugging:
char *name; // Name of lock.
struct cpu *cpu; // The cpu holding the lock.
uint pcs[10]; // The call stack (an array of program counters)
// that locked the lock.
};

當cpu要執行某段程式時，可以使用acquire(&listlock)上鎖，避免有兩個process同時執行那一段程式，要解鎖時可以用release(&listlock)解鎖，並清除debugging field。

acquire function的介紹:





















void acquire(struct spinlock *lk)
{
pushcli(); // 暫停 interrupt 的功能避免deadlock的發生
if(holding(lk)) //判斷CPU目前有沒有lock
panic("acquire");

// xchg是一個function，連續執行兩個步驟:
//1.判斷lock有沒有被佔有
//2.有的話就繼續等待，沒有的話就把lock設成1
//這樣的設計可以避免兩個process同時執行完判斷式後，都接著把lock設成1，造成race condition。

while(xchg(&lk−>locked, 1) != 0)//一直重複執行loop直到lock變成0(0代表未被其他process佔有)
;

__sync_synchronize();//不讓compiler重新安排程序，避免 deadlock

// Record info about lock acquisition for debugging.
lk−>cpu = mycpu();
getcallerpcs(&lk, lk−>pcs);
}

release function的介紹:
















void release(struct spinlock *lk)
{
if(!holding(lk)) 
panic("release");

lk−>pcs[0] = 0;
lk−>cpu = 0;

__sync_synchronize(); //不讓compiler重新安排程序，避免 deadlock

asm volatile("movl $0, %0" : "+m" (lk−>locked) : );

popcli(); //重新啟動interrupt的功能

}

Sleep lock:

有時候當程式需要長期持有lock的時候，我們會希望CPU能先去執行其他thread，而這時候我們就需要一個可以在不同thread之間都可以運作的lock，Xv6把這種lock叫做sleep-lock。
sleep lock不會把interrupt關掉，在等待的時候會yield processor，這麼一來可以確保其他的thread可以繼續跑。











void acquiresleep(struct sleeplock *lk)
{
acquire(&lk−>lk);
while (lk−>locked) {
sleep(lk, &lk−>lk);
}
lk−>locked = 1;
lk−>pid = myproc()−>pid;
release(&lk−>lk);
}

Deadlock:

Deadlock 發生在當A程序使用了lock X, B程序使用lock Y，而接下來A程式需要用到lock X，B程式需要用到lock Y，這樣就會造成A程序跟B程序都在等對方目前持有的lock，但卻一直等不到，形成Deadlock。為了要解決這樣的問題，所有的程序在獲得lock的順序就必須一致。
