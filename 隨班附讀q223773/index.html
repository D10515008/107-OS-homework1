Chapter 0 Operating system interfaces

0.1 前言

0.1.1 重點

  • OS Main Job
    1. 提供多個 process 比硬體更多有用的服務與資源。
    2. 管理和抽象低階硬體。
    3. 提供多個 process 可同時 (或看起來同時) 運行並使用硬體資源。
    4. 控制多個 process，使process之間可以共享數據及合作。
  • kernel 使用CPU的硬體保護機制，確保 user process 只能用自己的memory。
  • 只有kernel擁有這些硬體操作權限。
  • 當 user process 調用 kernel 時， kernel 會提升這些權限並執行預定義功能。
  • xv6中所有system call:

0.1.2 名詞解釋

  • kernel: 一個特殊的 program 提供服務給 process.
  • process: 運行中的 program，擁有自己的 memory (含指令、data、stack)
  • instruction: 指令，實現 program 的運算
  • data: 數據，運算時的變量
  • stack: 棧，function call.
  • system call: shell invokes kernel service.

0.2 Process and Memory

0.2.1 重點

  • process contains 2 parts:
    • user-space memory (指令、數據、棧).
    • per-process state private to kernal.
  • xv6 can time-share processes: 在CPU間不斷切換，決定哪個等待中的 process 被執行。when a process wait to excute, xv6 save its CPU registers, restoring them when it next runs the process.
  • a pid <-> a process.
  • child and parent 有各自的 memory 和 registers，改變變量並不會影量另一個process.
  • xv6 分配 user-space memory implicitly:
    • fork: allocate child’s memory to copy of parent’s memory.
    • exec: allocate enough memory to hold executable file.
  • xv6 沒有用戶概念，以 unix 角度就是都用 root 執行。

0.2.2 System call 說明

  • fork: 一次調用兩次返回(parent: return child pid, child: return 0)
  • exit: 停止當前process運行，並關閉資源(ex. memory and open files)
  • wait: return (當前process)已退出的child process pid，若沒有會一直等到有一個child退出為止。
  • sbrk: return location of new memory. when process needs more memory in runtime, call sbrk(n) to grow n bytes.
  • exec: replaces the calling process’s memory with a new memory image loaded from a file stored in the file system.
    • file should contain which part hold the instructions, which part is data, at which part instruction to start.
    • xv6 use ELF format.

0.3 I/O and File descriptors

0.3.1 重點

  • file descriptor(fd):
    • a fd <-> a “file”
      • a “file” could be 文件、目錄、device、pipe、copy of fd.
    • a small integer as a kernal object that process can read or write, like stream bytes.
    • every process has a private space of fd starting at zero.
    • console default’s fd (always open):
      • standard input: 0
      • standard output: 1
      • standard error: 2
    • 透過 fd 做 I/O 重定向.
  • ls file1 file2 > tmp.txt 2>&1
    • file1 file2 的錯誤輸出都會出現在 tmp.txt
    • 2>&1 : fd2 is dup of fd1.

0.3.2 System call 說明

  • read(fd, buf, n): return 讀取的字節數(最多為n), n=0讀取結束.
  • write(fd, buf, n): return 寫入的字節數(不為n是發生錯誤).
  • close(fd): free the fd, so that open, pipe, dup can reuse.

0.4 Pipes

0.4.1 重點

  • pipe
    • a kernel buffer
    • a pair of fd, one for write, one for read.
    • create process tree, leaves are command, process waits for left and right both complete.
  • pipe 比 no pipe 更好:
    1. clean tmp file auto.
    2. pass arbitrarily long streams of data.
    3. allow for parallel execution of pipeline stages.
    4. in inter-process communication, pipes’ blocking reads and writes are more efficient than the non-blocking semantics of files.

0.5 File System

0.5.1 重點

  • data file(file): uninterpreted byte arrays
  • directory(dir): named referance to file or dir.
  • root(/): 根目錄, form a tree.
  • current directory(.): 當前目錄.
  • xv6 所有對 file system 的操作都是實作成 user-level program，使得 shell 容易被擴充，但 UNIX 時代是將這些command建在 shell 裡 (並且將 shell 建在 kernel).
  • cd 是例外，它必須建在 shell 裡. 若是做為一般 command 會變成 fork child process，改變 child process 的當前目錄並不會影響到 parent process 的當前目錄.

0.5.2 System call 說明

  • chdir(dirname): 改變當前目錄.
  • mkdir(dirname): 當前目錄下建新目錄.
  • open(filename, O_CREATE): 開新文件.
  • mknod(name, major, minor): 建新 device 文件.
    • device 文件
      • 沒有文件內容
      • 透過 metadata as a device file, and major and minor device numbers define a kernal device.
      • open device file 是將讀寫直接指向 kernal device 實現，而非指向 file system.
  • fstat(fd): return inode.
    • 實際文件 data structure
      • uniqe inode number
      • 可以有多個名字
      • inode and 磁碟空間是當 nlink=0 時才會清空。
  • link(f1, f2): 對 f1 指向的文件新建另一個新名稱 f2.
    • link(‘a’,‘b’): 如果後續對a讀寫, 由於b也是指向同一文件所以也同樣是對b讀寫
  • unlink(f1): 移除 f1 對文件的指向.

0.6 Real World

0.6.1 重點

  • UNIX 整合"標準的" fd、pipe、shell command 在一起，這種通用與可重用想法促使發展強大的"軟體工具"文化。
  • Portable Operating System Interface (POSIX) standard 是 UNIX system call 的標準. xv6 並沒有服從POSIX標準，只實作了部分 (ex. 沒有 lseek ).
  • morden kernals 發展快速，usually are beyond POSIX standard，提供更多的服務。
  • 文件系統抽象的概念被應用在 WWW 網路資源.
  • Multics 複雜的設計讓 UNIX 的設計者想把文件系統寫的更簡單.

Chapter 1 Operating system organization

1.1 前言

1.1.1 重點

  • os start new process with fork.
  • os must time-share amnog processes.
  • os 必須確保 process 的 isolation.
  • xv6 只適用 intel 80386 及往後的 x86 架構.

1.2 Abstracting physical resources

1.2.1 重點

  • embedded devices or real-time systems 可以直接與硬體溝通.
  • strong isolation 設計可避免直接使用敏感性硬體資源，也可避免 processes 之間因為 bug 而 crash 最後彼此干擾.
  • time sharing 設計可避免單一 process 占用cpu.

1.3 User mode, kernel mode, and system calls

1.3.1 重點

  • kernel mode: 允許 processor 執行 privileged instructions.
    • ex. read/write disk
  • user mode: application 只被允許執行 user-mode instructions, 並且只在 user space 運行.
    • ex. adding numbers
  • kernel: software running in the kernal space.
  • application 若要執行 privileged instructions, 必須轉換到 kernel mode at an entry point, 由 kernel 檢驗這些 system call 是否合法執行。

1.4 Kernel organization

1.4.1 重點

  • monolithic kernel: 整個 OS 放在 kernal, 包括所有 system call 實作都在 kernel mode.
    • 優: 不必區分哪些不需要 hardware privilege, 也容易讓 OS 之間更容易協作, 例如一個 buffer cache 可以同時分享給 file system 和虛擬記憶體使用.
    • 缺: 如果發生錯誤，直接當機.
  • microkernel: 大部分放在 user-level, 只保留低階 function 給啟動、發訊息、連接 device 等. 將大部分放在 user-level，可減少因為發生錯誤直接當機的風險。
  • 大部分的作業系統都是混和模式.

1.5 Process overview

1.5.1 重點

  • page table: 由硬體實作，將虛擬地址對應至實體地址.
  • xv6 使用頁表來提供 process 各自的虛擬地址.
  • xv6 從 0x80100000 開始，0-0x80000000 是給 user-level 使用.
  • process 運行時 kstack 為空，processor 使用 state；process 被中斷或進行 system call 時 processor 會改用 kstack 而 state 保存數據不使用. thread 會在 kstack 與 state 之間切換.

1.5.2 資料結構

  • proc ( p ) : process
  • p->kstack: kernel state.
  • p->state : process state (user-level).
  • p->pgdir : process page table.






1.6 Code: the first address space

1.6.1 重點

  • PC 開機時會將 boot loader 從硬碟載入記憶體，entry 開始執行，此時 paging hardware 還沒開始工作.
  • 0xa0000:0x100000 is for I/O devices.
  • kernal starts at 0x100000.
  • KERNBASE=0x80000000.
  • 限制: kernel instructions and data 必須在 4mb 以內.
  • entry: 執行設定 page table 工作.
    • 運行中，將虛擬地址 0:0x400000 映射到實體地址 0:0x400000.
    • 結束後，將虛擬地址 KERNBASE:KERNBASE+0x400000 映射到實體地址 0:0x400000.
  • entrypgdir 的實體地址放入暫存器 %cr3
    • V2P_WO - KERNBASE 就是 entrypgdir 的實體地址 (因為 entrypgdir 指向高處)
    • %cr0=CR0_PG，設定 paging hardware 開始工作
  • entry 轉到 kernel’s C code, 必須設置在高位處 (KERNBASE) 執行 kernal (因為 PC-relative direct jump would execute the low-memory version of main).

1.6.2 資料結構

  • main

1.7 Code: creating the first process

1.7.1 重點

  • userinit 只有在第一次新建 proc 才有被呼叫到.
  • allocproc 每次新建 proc 都會被呼叫到.
  • allocproc 會去查 proc table 找到 state 為 UNUSED 的 proc, 然後將 UNUSED 改成 EMBRYO 並給它 pid; 然後嘗試分配 kernal stack, 如果分配失敗, allocproc 會改回 UNUSED 並 return 0 表示失敗.

1.7.2 資料結構

  • allocproc

1.8.1 重點

  • allocproc -> forkret -> trapret

  • mpmain

1.9 The first system call: exec

1.9.1 重點

  • Initcode.S start
    1. 將 $argv $init $0 三個值 push to stack
    2. %eax=$SYS_exec
    3. int $T_SYSCALL 告訴 kernel 執行 exec
  • 如果運行正常，exec 不會返回，並調用 init
  • 如果運行失敗，exec 會返回，並調用 exit

1.9.2 資料結構

1.10 Real world

1.10.1 重點

  • real os find free proc with explicit free list in constant-time search, but xv6 in linear-time search in allocproc.
  • xv6 在地址空間規劃上有缺點，無法使用超過2GB實體記憶體.
