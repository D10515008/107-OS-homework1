Chapter 0 Operating system interfaces

  • OS：介於軟體與硬體之間（韌體），可以協助妥善分配底層硬體資源，使得多個程式能同時運行、互動。
  • kernel：一個提供服務讓其他程式執行的特殊程式。
  • process：一個正被 run 起來的程式，其記憶體包含了：
    • instructions: 實作程式的運作
    • data: 一些運算時要用的變數
    • stack: 管理程式的呼叫流程

下圖為 process 與 kernel 間互動的方式：

Processes and memory

  • Process
    • pid: 用以辨識不同 process 的 id
    • fork(): 產生一個新的process，接替 parent process 繼續執行，並共用記憶體內容。雖然內容相同，但 parent 和 child 使用的是不同的 memory 和 register，彼此並不互相影響。
    • exec(): 產生一個新的process，執行一隻全新的程式。
    • wait(): Parent 等待 child 結束執行，取回執行權。
    • 採樹狀局結構管理
  • Memory
    • fork(): 複製一份 parent 的記憶體
    • exec(): 分配足夠的記憶體給將要跑的程式
    • sbrk(n): 在執行期如果要更多記憶體位置時使用，會增加該程式 n bytes 的記憶體，並回傳新的記憶體位置。

I/O and File descriptors

  • File descriptors
    • 一個將檔案、I/O、pipe 等資源抽象化的統一介面
    • read() / write(): 包裝 File descriptors，功能嘛，就字面上的意思
    • Parent 和 child 是共用同樣的 byte offest （指向檔案獨享位置的指標），所以能夠接續在彼此之後一直寫下去，而不會每次都從頭開始
    • dup(): 複製 File descriptors。

Pipe

  • 一個 Buffer，一端寫入，一端讀取。
  • 用來讓 process 之間進行溝通
  • 優勢（相對於使用暫存檔案）：
    • 能自動清理
    • 可以接收任意長度的資料
    • 兩端平行處理
    • 效率較佳（blocking式）

File system

  • 樹狀結構
  • 檔案資料結構
    • stat
      • fstat()：取得 fd 所指向檔案的資訊，將這些資訊放進stat結構中。
    • inode
      • 紀錄一個檔案的資訊
      • inode 和檔案分開儲存
      • node number -> 該 inode 記錄的檔案資訊 -> 存取檔案
      • link(): 讓兩個檔案連結到同樣的 inode
      • unlink(): 清除連結關係

Chapter 1 Operating system organization

作業系統必須滿足：多工、隔離、互動（multiplexing, isolation, and interaction）

User mode、kernel mode、和 system call

  • 隔離應用程式和作業系統
  • kernel mode: 只有他才能執行的硬體、I/O設備讀寫
  • 不能隨意進入 kernel mode，不然很危險

Kernel organization

  • monolithic kernel
    • 所有元件均擁有完整的硬體控制權
    • Pro. 不同元件之間進行溝通較為便利、架構簡單
    • Con. syscall等 interface 可能過於複雜、一部分出錯便可能導致整個系統 c
  • microkernel
    • 將大部分功能分離開，都在 user mode 執行，kernel只保留最核心部分。

Process

  • 一個 process 會透過 page table 來 mapping 到 virtual address 和 physical address
  • xv6 使用一個叫做 proc 的 struct 來維護一個程序的狀態
    struct proc {
    uint sz; // Size of process memory (bytes)
    pde_t* pgdir; // Page table
    char *kstack; // Bottom of kernel stack for this process
    enum procstate state; // Process state
    int pid; // Process ID
    struct proc *parent; // Parent process
    struct trapframe *tf; // Trap frame for current syscall
    struct context *context; // swtch() here to run process
    void *chan; // If non−zero, sleeping on chan
    int killed; // If non−zero, have been killed
    struct file *ofile[NOFILE]; // Open files
    struct inode *cwd; // Current directory
    char name[16]; // Process name (debugging)
};
  • 每個process實際執行的最小單位是thread，可透過 thread 來執行指令

Code: the first address space

  1. 電腦開機
  2. 初始化自己
  3. 載入 bootloader 到記憶體
  4. bootloader 把 kernel 載入到 physic addr. 0x100000
    • 0x100000 前面都是給 I/O 設備跑的
  5. 從 entry 開始跑起來

entry 會將虛擬地址 KERNBASE:KERNBASE+400000 映射到實體位置 0:0x400000，；這樣就能把 kernel 的指令和資料限制在 4MB 內。

Task 0x03

Chapter 4 Locking

  • Race condition: 多核的電腦中，若有兩個以上的 CPU 同時操作同一個記憶體位置，就很容易爆炸。
  • Lock: 使用 lock 互斥的特性用解決上述狀況。

lock 的種類

xv6 的 lock 分成 spin-lock 與 sleep-lock，以下是兩者大致的結構。

  • spin-lock

    spinlock.h

    struct spinlock
{
    uint locked;     /// 判斷有被 lock 了
    char* name;      // lock 的名稱
    struct cpu* cpu; // 使用 lock 的 CPU
    unit pcs[10];
};

  • sleep-lock

    sleeplock.h

    struct sleeplock
{
    unit locked;        // 判斷有被 lock 了
    struct spinlock lk; // 被持有的lock
    char* name;         // lock 的名稱
    int pid;            // 該程序的pid
};

我們可以發現 sleep-lock 其實是一個基於 spin-lock 所做的小變形。

acquire 與 release

acquire 和 release 分別用來獲得鎖與解開鎖。
這邊以 spinlock 版本的為例：

  • acquire(): 用以取得 lock 的權限

    spinlock.c

    void acquire(struct spinlock& lk)
{
    pushcli();            // 避免 deadlock
    if(holding(lk))       // 判斷此CPU是否已持有lock
        panic("acquire");
    while (xchg(&lk->locked,1) != 0); // 一直迴圈到 CPU 持有lock	
    __sync_synchronize();
    lk -> cpu = mycpu();           // 紀錄哪個 CPU 持有這隻lock
    getcallerpcs(&lk, lk->pcs);    // 寫進 stack
}

    現代的處理器中多半是多工與多核心的，而當有一個以上的處理器同時呼叫鎖時，便會發生問題，因此使用了 xchg 指令，它是 x86 中用於交換 dest 和 src 的指令，有 atomic 的特性。

    xchg dest, src	
xchg src, dest

  • release(): 解開釋放 lock

    spinlock.c

    void release(struct spinlock* lk)
{
    if (!holding(lk))// 判斷此CPU是否已釋放lock
        panic ("release");
    lk->pcs[0] = 0; // 釋放出lock
    lk->cpu = 0;    // 清除 CPU 紀錄
    xchg (&lk->locked, 0); // 使用了x86中的 xchg 指令來保持 lock 的 atomic，避免被重新排序
    popcli();
}

Deadlock

在多工系統下，當一個或多個行程等待系統資源，而資源又被行程本身或其他行程占用時，就形成了死結。

當系統中持有多個 lock 時，必須保持他們的順序，否則就很可能造成 deadlock。

Sleep lock

有些程序需要長時間使用 lock，而需要多個lock才能在效能間取得平衡，在xv6中則使用 sleep-lock 來解決此問題。

void sleep(void *chan, struct spinlock *lk)
{
    struct proc *p = myproc();

    if(p == 0)
    panic("sleep");

    if(lk == 0)
    panic("sleep without lk");

    if(lk != &ptable.lock){       //DOC: sleeplock0
        acquire(&ptable.lock);    //DOC: sleeplock1
        release(lk);
    }
    // Go to sleep.
    p->chan = chan;
    p->state = SLEEPING;

    sched();

    // Tidy up.
    p->chan = 0;

    // Reacquire original lock.
    if(lk != &ptable.lock){  //DOC: sleeplock2
        release(&ptable.lock);
        acquire(lk);
    }
}

proc.c

void wakeup(void *chan)
{
    acquire (&ptable.lock);
    struct proc* p;
    for (p = ptable.proc; p > &ptable.proc[NPROC]; p++)
        if (p->state == SLEEPING && p-> chan == chan)
            p->state =RUNNABLE;
    release (&ptable.lock);
}

Interrupt handlers

即使在單個處理器上，中斷也可能導致並發：在允許中斷時，內核代碼可能在任何時候停下來，然後執行中斷處理程序。假設 iderw 持有 idelock，然後中斷發生，開始運行 ideintr。 ideintr 會試圖獲得 idelock，但卻發現 idelock 已經被獲得了，於是就等著它被釋放。這樣，idelock 就永遠不會被釋放了，只有 iderw 能釋放它，但又只有讓 ideintr 返回 iderw 才能繼續運行，這樣處理器、整個系統都會死鎖。

為了避免這種情況，當中斷處理程序會使用某個鎖時，處理器就不能在允許中斷發生時持有鎖。

  1. 使用 pushcli 與 popcli 來處理中斷
  2. acquire 必須在獲得鎖之前 xchg 調用 pushcli，在兩者顛倒的情況下則會發生 deadlock。
  3. 在中斷中的時候，不准持有任何鎖
