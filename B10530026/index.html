Chapter1

User mode , kernel mode , and system call

在x86的處理器架構之下，有所謂的user mode 及 kernel mode這兩種模式。
而在user mode時，處理器通常不被允許直接執行底層的程式與指令，此時必須透過system call等間接的指令或進入kernel mode去執行。
這種機制是為了阻絕被惡意程式或單一程式直接吃掉大量處理器資源的方法。

Kernel Organization

  • monolithic kernel (整塊性核心)
    一如名字，這種kernel的模式是整塊作業系統都塞入硬體裡面。
    好處是架構簡單，執行方便，作業系統(不分任何部分)都擁有完整的硬體控制權限。
    壞處是在不同部分的作業系統之間的介面(interface)常常是複雜的，在指令相互傳遞時
    容易出錯，而一旦出錯，將會造成整個kernel停止運行，當kernel停止運行，電腦將會
    停止運作且所有的應用程式將直接爆炸。此時，電腦就需要重新開機了。

  • microkernel (微內核)

    與monolithic kernel相反的是，微內核(microkernel)所強調的是將作業系統上的大部分功能安裝在 user mode上，當有需要時，在 user mode 上的 shell 再經由 microkernel 回傳 message 給user mode端的File server

  • 然而，在現實世界裡，這類二分法並非絕對。
    Linux就是有著monolithic kernel，但在其之上的作業系統的函式(function)卻是在user mode 執行。

Process Overview

為了讓每個process都認為自己擁有完整的系統控制權，以及認為自己可以"獨佔"的一顆CPU，xv6使用pagle table來在virtual address（process認為自己可以"獨自"使用的記憶體位置）和physical address（CPU實際可以在硬體上操作的部分）之間轉換

每個uesr端的process的address space從0開始，由下往上分別是程式資料和global變數、stack和heap；再往上則是kernel的指令和資料，將kernel也一同mapping到這個 address space的優點是可以在執行system call時可以直接往上參照個別process的資料。

每個process實際執行的最小單位是thread（一個process可能有好幾個thread），而xv6為每個process建立了一個叫做proc的struct結構用以記錄該process的各種狀態(如running ,waiting ,ready之類的)。

當一個process執行一個system call，processor會切換到kernel stack去執行privileged instructions好讓system call被實施(implement)，而當system call完成之後，processor又會切換回user stack，依此反覆。

  • p->state 指出這個process是什麼狀況，例如allocated, ready to run, running, wait-ing for I/O, or exiting。
  • p->pgdir 保留這個process的page table，在執行這個process的過程中。

The First address space

當電腦開機之後，他會先初始化自己，然後從硬碟將boot loader載入至記憶體，接著boot loader再將kernel從硬碟中讀取出來，並且從entry的位置開始執行。
這時候的x86 的paging hardware還沒運作，所以virtual addresses 還是直接映射到physical addresses的。

接著，為了讓kernel剩餘的部分能夠正常運作，entry會建立一個page table，將虛擬記憶體0x80000000（KERNBASE）平行移動(map)到實體記憶體0x0，最後是將KERNBASE+0x400000平行移動到0x400000，這樣，kernel的指令和資料空間就會限制在4 Mbytes之內。

Create the First process

在初始化之後，系統通過調用userinit建立了第一個process。接著userinit會調用allocproc。
allocproc的工作是在page table中分配一個槽（即結構體struct proc），並初始化process的狀態，然後為kernel thread分配stack空間並為該process創建專屬的pid。
注意一點：userinit僅僅在創建第一個process時被調用。

在xv6中，user mode和kernel mode之間的切換(switch)是透過interrupt達成的，因此要進行切換的時候，系統會將user mode中所使用到的register中的資料存放到kernel stack中，這樣要return時就可以取回資料繼續執行。

一般來說，程式的執行順序：讀取指令->更新program counter->讀取指令…如此反覆下去，但有時候需要由作業系統（kernel）介入，類分以下三種：

1. system call 程式要求執行一些由系統提供的服務。
2. exception   比方說除以0或是存取page table以外的記憶體空間等非法(illegal)操作
3. interrupt   例如clock cycle定期引發interrupt以達成time sharing

以上所共通的是，當程式執行時暫存器所儲存的資料都必須存取起來好讓程式可以中斷之後回來繼續執行下去。例如system call參數以及kernel執行的開始位置等。
注意一點，雖然interrupt和trap這兩個詞會交替使用，但trap是由目前proceeor執行的process發出的。而interrupt則是由hardware所產生，可能與process無關。

X86的保護機制

x86有四種保護機制由0（權限最高）到3（權限最低），在實際使用中，大部分的作業系統只會使用到兩種權限級 0跟3，分別對應要kernel mode和user mode。執行中指令的權限等級（current privilege level）會存放在%cs暫存器中的CPL欄位中。

在x86 中，中斷處理程序的入口在中斷描述符表（IDT）中被定義。這個表有256個表項，每一個都提供了相應的%cs 和%eip。

interrrupt descriptor table（IDT）則記錄了在不同的interrupt情況下，%cs和%eip暫存器該如何操作。

發生system call時，程式會執行 int 指令，過程描述如下：

  • 從IDT中取得第n筆descriptor，n是int執行的參數
  • 檢查%cs中的CPL是否<=DPL，DPL是前述取得descriptor的privilege level
  • 若DPL<CPL則將%esp和%ss的值儲存在CPU的內部暫存器中
  • 從前述的descriptor中讀取%ss和%esp
  • push %ss
  • push %esp
  • push %eflags
  • push %cs
  • push %eip
  • 若是interrupt的話則清除%eflags中的IF bit
  • 將%cs和%eip設為descriptor中的值

作業系統會使用 iret 指令來從一個 int 指令中return。它會將 int 過程中存進stack的值pop出來，然後恢復%eip的值讓程式依照原本的順序繼續執行。

圖3-1 代表執行過一個int指令後的情況，這是發生了權限級轉換（即描述符中的權限級比CPL 中的權限級的時候）stack的情況。如果這條指令沒有導致權限級轉換，x86 就不會保存%ss 和%esp。在任何一種情況下，%eip 都指向中斷描述符表中指定的地址，這個地址的第一條指令就是將要執行的下一條指令，也是int n 的中斷處理程序的第一條指令。作業系統應該實現這些中斷處理程序。

The first system call

system call的參數，argint、argptr、argstr 和 argfd這幾個函式分別用於取得int、pointer、string和file descripot類型的參數。其中%esp會指向system call的return address，參數剛好就在%esp+4的地方，因此第n筆參數的位置就是%esp+4+4*n。

Assembly trap handlers

x86總共有256種不同的interrupt，其中0-31是軟體的interrupt（除以0或是存取非法記憶體位置等）、32-63是硬體的interrupt，最後64定義為system call。

處理器處於user mode時，從task segment descriptor讀取%esp和%ss，將原本的%esp和%ss存入stack中。
若是處於kernel mode，則以上操作皆不發生。接著push %eflags、%cs和%eip進stack，對於page fault等種類的interrupt則連錯誤資訊一並存入，最後從相應的IDT讀取%eip和%cs。

剩下的工作會交給 Alltraps 完成；push %ds、%es、%fs和%gs，以及其他的通用暫存器。這時kernel stack就會有一個完整的 trapframe 結構，好讓系統在執行權從kernel交回給原本的process時，能保有和當初相同的工作狀態。

Interrupts
interrupt某些部份和system call很類似，但不同之處是system call通常是根據固定的程式邏輯發出的，但interrrupt有可能是由硬體裝置在任何時間所產生。
舉個例子，主機板上的timer裝置每100ms發出一次interrupt，讓系統可以記錄時間和執行time sharing，但這樣的頻率又不會高到過於干擾系統運作。
像x86 處理器一樣，PC 主板也在進步，並且提供中斷的方式也在進步。早期的主板有一個簡單的可編程中斷控制器（被稱作PIC），你可以在picirq.c 中找到管理它的代碼。

隨著多核處理器主板的出現，需要一種新的處理中斷的方式，因為每一顆CPU 都需要一個中斷控制器來處理髮送給它的中斷，而且也得有一個方法來分發中斷。這一方式包括兩個部分：第一個部分是在I/O 系統中的（IO APIC，ioapic.c），另一部分是關聯在每一個處理器上的（局部APIC，lapic.c）。xv6 是為搭載多核處理器的主板設計的，每一個處理器都需要process接受中斷。

Disk driver

驅動程序是作業系統中用於管理某個設備的程式：它提供設備相關的中斷處理程序，操縱設備完成操作，操縱設備產生中斷，等等。驅動程序可能會非常難寫，因為它和它管理的設備同時在並發地運行著。另外，驅動程序必須要理解設備的接口（例如，哪一個I/O 端口是做什麼的），而設備的接口又有可能非常複雜並且檔案稀缺。

–
kernel在系統開機的時候呼叫 ideinit 來初始化disk driver（對IDE硬碟而言），接著ideinit會再呼叫 ioapicenable 來發出IDE_IRQ的interrupt訊號。

接著ideinit會開始檢查硬體，它會先呼叫 idewait 讓硬碟準備好接收指令。接著主機板會將硬碟的status bit呈現在0x1f7的I/O腳位上，然後ideinit會在busy bit (IDE_BSY)被清空和ready bit (IDE_DRDY)被設定好之後接收前述的status bit。

ideinit之後，就只能透過buffer cache去調用 iderw ，iderw會根據buf結構中flags的值來決定該對硬碟採取何種讀寫策略：B_DIRTY被設為1的話代表要將buffer的內容寫入硬碟；B_VALID是0的話則將資料從硬碟讀入buffer。

硬碟的讀寫相較CPU而言是很花時間的，這裡所花的時間始以毫秒級計算，相同的，一直查看硬碟的讀寫狀態也很浪費時間和資源；因此iderw採用的策略是讓其他process享有CPU的資源、等到讀寫完成等事件發生再透過interrupt通知CPU，這時候需要使用一個queue的結構來記錄那些硬碟操作還在等待中。

iderw會把新進入的buffer資料加入queue的尾端，如果一個buffer已經排到queue的最前頭了則會呼叫 idestart 來進行實際的硬碟操作，否則要等到前面的buffer都已經被處理完了才會輪到自己。
注意的是，iderw有一些關於IDE設備的細節，並且在幾個特殊的端口進行讀寫。如果任何一句outb語句錯誤了，IDE就會做一些我們意料之外的事。保證這些細節正確也是寫設備驅動程序的一大挑戰。

將讀寫的request加入queue之後iderw就暫時沒事可做了，必須等待結果揭曉。因此iderw此時會開始進入睡眠狀態，等待操作結束之後interrupt handler再修改flags的值。這時CPU就可以將資源分配給其他process以免浪費運算資源。

最後讀寫終於完成、並且觸發interrupt之後，trap會呼叫 ideintr 接手剩餘的工作。此時ideintr會詢問queue裡的第一個buffer，藉以得知方才完成的操作是讀還是寫，如果是讀取、disk controller中也有資料正在等待的話，ideintr會使用 insl 將資料讀進buffer；最後設定B_VALID、清空B_DIRTY、喚醒其他正在等待的process，最後把下一個等待中的buffer交給disk。
