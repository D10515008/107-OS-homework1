項目二：Chapter 0 and 1

Ch.0

OS工作於應用程式與硬體之間，負責資源的管理，有了OS，可以讓process共享硬體資源，process也不用管細部的硬體處理流程，只要透過OS提供的interface即可。OS的核心為kernel，一般的process運作於user mode，並沒有權限可以操作硬體，當需要操作硬體時，會呼叫system call，進而提升權限進入kernel mode，進入kernel mode後，就有權限可以進行所有硬體的操作。以下為xv6有的system call

Processes and memory
xv6可以進行分時處理，讓多個process同時執行，每個process利用pid來識別，當執行的process切換時，會進行context switch，將目前的執行狀態存起來，再次執行時，再將狀態重載回來。fork可以用來建立一個child process，parent和child擁有相同的記憶體內容，但執行時使用不同的記憶體和暫存器。exec可以呼叫另一支程式，並給他執行的參數。

I/O and File descriptors
File descriptor為一個整數，代表要從哪個物件讀或寫，0代表standard input，1代表standard output，2代表standard error，也可以透過open,pipe或dup來取得file descriptor。File descriptor可以用來做I/O的redirection。

Pipes
Pipe是小型的buffer，可以用來讓不同的process溝通，例如把一個process的stdout導到另一個process的stdin，pipe比temporary files有4種優勢，包括不用另外刪除temporary files、pipe可以傳任意長度的data streams、pipe允許parallel execution和pipe的blocking reads and writes比tempprary files的non-blocking更有效率。

File system
xv6的file system為樹狀結構，/為起始，稱為root，.代表目前的目錄，..代表上一層目錄，~代表家目錄。在unix-like的系統中，所有東西都被視為file，不論是directory或是device都一樣。link可以用來建立一個連結指到同一個inode，而unlink可以解除連結。

Ch.1

OS必須分時共享資源，為了確保每個process都能正常運作，因此需要將process隔離開來，當一個process有bug或fail時，才不會影響到其他process。OS必須滿足multiplexing, ioslation和interaction這三個要求。

Abstracting physical resources
為了達到ioslation，禁止直接操作硬體資源是有幫助的，因此透過system call interface的方式，來達到abstracting physical resources，讓process可以共享資源。

User mode, kernel mode, and system calls
OS和應用程式之間，必須有一個分界，當應用程式出錯時，才不至於影響整個OS的運作，所以process執行時分為user mode和kernel mode，user mode沒有權限執行系統的操作，而kernel mode有，當運作於user mode的process要進行系統操作時，就必須呼叫system call，進到kernel mode來執行。

Kernel organization
將所有的system call運作於kernel mode，這種架構稱為monolithic kernel，因為操作系統時，擁有完整的權限，因此有利於OS不同部分的合作，但缺點就是會非常複雜，OS開發者會很容易出錯，而在kernel中，出錯是非常致命的，為了降低出錯的風險，OS開發者會盡少讓OS的code執行於kernel mode，盡可能執行於user mode，這種架構稱為microkernel。

Process overview
為了加強process間的隔離，process abstraction讓每個process像是一個私人的機器，擁有自己的記憶體和空間，而且其他process不能存取。xv6的虛擬記憶體，利用page table來將virtual memory address對應到physical memory address上。xv6維護每個process的state，並聚集到struct proc中，包括page table,kernal stack和run state。

項目三：Chapter 6

File system用來儲存資料，也支援使用者和應用程式間資料的共享，這些資料，在重開機後必須保留。xv6的file system提供Unix-like的files,directories和pathnames,並將資料存在disk上保存。這樣的file system有以下幾種挑戰:

1.File system需要on-disk的資料結構來代表樹狀結構，並記錄哪個block是有資料的，哪個block是空的。
2.File system必須支援crash recovery，當crash發生時，file system在重啟後，必須正確的執行。
3.不同的process可能同時存取file system，因此需要確保正確的存取。
4.從disk存取資料遠比從memory慢，因此需要為常存取的block維護一份in-memory cache。

Overview
xv6的file system如下圖所示分為7層，disk layer從硬體上讀寫blocks，buffer cache layer caches disk blocks和確保同一時間只有一個process可以更改block裡的資料，logging layer允許上層在一個transaction中，更新好幾個block和確保crash前，block都以atomically的方式更新，inode layer提供檔案辨別，每個檔案具有唯一的i-number和哪些blocks存有這個檔案的資料，directory layer將目錄實作成一種特殊的inode，內容是目錄的串列，每個目錄包括檔案名和i-number，pathname layer提供分層的路徑名，file descriptor layer將許多Unix資源抽象化，包括pipes,devices和files。

下圖是inode和blcok於disk的位置圖，blcok0為開機相關的資料，block1為superblock，存放關於file system的metadata，block2放log，之後為inodes、bitmap和datas。

Buffer cache layer
buffer cache layer的主要工作為同步對disk blocks的存取和cache常用的blocks，避免重複到比較慢的disk拿資料。主要的interface有bwrite、bread和brelse，為了避免多個process同時使用同樣的buffer cache，當bwrite或bread時，會lock buffer，沒用的時候再利用brelse來release lock。buffer cache的長度有限，buffer cache會優先回收最少使用的buffer。

Code: Buffer cache
buffer cache的資料結構為doubly-linked list，main會call binit來初始化，可以透過bcache.head來取得buffer cache的linked list。buffer有兩個狀態位，B_VALID表示buffer裡有block的copy，B_DIRTY，表示buffer的內容被更改過，必須寫回disk。要讀取buffer時，Bread會call bget來取得buffer的內容，當資料不在buffer裡時，會call iderw來從disk要資料，而bget透過掃描buffer list的方式來找資料，找到資料後會對buffer進行sleep-lock並return buffer。要寫進buffer時，會設定B_DIRTY且call iderw來寫回disk。對buffer操作完後，會call brelse來release lock，並將buffer移到linked list的前面，因此linked list會照最近使用的時間排序。

struct {
  struct spinlock lock;
  struct buf buf[NBUF];

  // Linked list of all buffers, through prev/next.
  // head.next is most recently used.
  struct buf head;
} bcache;

Logging layer
xv6的system call並不直接寫入on-disk file system data structures，而是將要寫的放在disk的log裡，並記錄狀態，當commit時，再一次寫入on-disk file system data structures，當寫入成功，就會將log清掉。當系統crash重開後，如果log中有完成狀態的log，則會再次將它寫入disk，其他未完成狀態的log則直接清掉。

Log design
logging system可以累積很多write再一次commit，因此在單次commit中，會包括很多完整system call的write，為了避免system call跨transactions，所以logging system只有在沒有file system call執行時commit。

Code: logging
begin_op會等待，直到logging system沒有在commit和有足夠的空間來保存這次的寫入和正在執行的system calls。log_write紀錄block的sector number在memory，保留在disk的log中，並標記buffer的B_DIRTY。end_op會先遞減outstanding system calls的count，當count為0時，才會執行commit。recover_from_log在user第一個process啟動前由initlog呼叫，它會讀取log的header，並執行已經commit的transaction。

struct logheader {
  int n;
  int block[LOGSIZE];
};

struct log {
  struct spinlock lock;
  int start;
  int size;
  int outstanding; // how many FS sys calls are executing.
  int committing;  // in commit(), please wait.
  int dev;
  struct logheader lh;
};
struct log log;

Code: Block allocator
xv6的block allocator維護一個bitmap在disk中，每個block用1個bit來表示，0代表block沒被使用，1代表有被使用。balloc用來分配一塊新的block，它會從0開始找，直到sb.size，當找到時，會更新bitmap並return那個block。bfree用來釋放一塊block，它找到正確的bitmap位置後，就將那個bit清掉。

Inode layer
inode有兩種意義，一種是on-disk data structure，包含檔案大小和data block numbers的list，一種是in-memory inode，包含on-disk inode的copy和額外的資訊。on-disk inode在disk中是連續的，而且每個inode有相同的大小。in-memory inode只有在被C pointer指向時，才會在memory中，當reference count為0時，in-memory inode就會被踢出memory。

Code: Inodes
ialloc用來分配一個inode。iget從inode cache中找可用的entry，當找到時，會return對那個inode的新reference。iput透過減少reference count來release C的pointer。ilock用sleep-lock來lock inode，而iunlock用來releas sleep-lock。

// in-memory copy of an inode
struct inode {
  uint dev;           // Device number
  uint inum;          // Inode number
  int ref;            // Reference count
  struct sleeplock lock; // protects everything below here
  int valid;          // inode has been read from disk?

  short type;         // copy of disk inode
  short major;
  short minor;
  short nlink;
  uint size;
  uint addrs[NDIRECT+1];
};

Code: Inode content
下圖為file在disk中儲存的方式，on-disk inode的資料結構中，包含type,nlink,size和block number的array。

// On-disk inode structure
struct dinode {
  short type;           // File type
  short major;          // Major device number (T_DEV only)
  short minor;          // Minor device number (T_DEV only)
  short nlink;          // Number of links to inode in file system
  uint size;            // Size of file (bytes)
  uint addrs[NDIRECT+1];   // Data block addresses
};

Code: directory layer
directory就像file一樣，它inode的type為T_DIR，它的資料就是一連串的directory entry，每個entry都是struct dirent，包含名稱和inode number。dirlookup用來找目錄，找到後會return inode的pointer。dirlink用來寫入一個具有給定的名稱和inode number的新directory entry到目錄中。

struct dirent {
  ushort inum;
  char name[DIRSIZ];
};

Code: Path names
namei evaluate path和return相對應的inode。namex首先決定evaluation的起始位置，當path為/時，從root開始，否則從當前目錄，然後透過skipelem依序考慮路徑的每個元素，以iteration的方式在inode中尋找名稱。

File descriptor layer
Unix中，所有東西都可以視為file，包括devices和directory。所有的open files都會被記錄在global file table中。filealloc從file table中找unreferenced file，並return新的reference。filedup複製一個descriptor並增加reference count。fileclose減少reference count，當count為0時，就release掉。filestat,fileread和filewrite就是實作stat,read和write在file上的操作。

struct file {
  enum { FD_NONE, FD_PIPE, FD_INODE } type;
  int ref; // reference count
  char readable;
  char writable;
  struct pipe *pipe;
  struct inode *ip;
  uint off;
};
