項目二: 第〇、一章

作業系統是夾在軟體和硬體之間，讓程式能夠共享硬體，讓硬體能夠達到多工，且能讓使用者跟硬體溝通的介面。一個作業系統最重要的包含：

  • Kernel
    從shell接收system call，為執行的程式提供服務
  • Shell
    提供與使用者互動的UI介面
  • IO / File descriptor
    接收外部的輸入輸出
  • File system
    管理和組織檔案之間的關聯

作業系統要達到多工的話，需要讓各個程序間達到 multiplexing, isolation, 和 interaction 等等需求。最主要的做法，就是讓程式只能使用 user mode 的指令，透過 kernel 溝通，讓 kernel 有最高管理權限防止惡意程式直接操控硬體；另外， kernel 也要幫程式分割 address space ，讓他們只能在各自的記憶體工作，達到 isolation 的作用。 Kernel 使用 stack 來管理這些 address space ，並使用 page table 管理實體記憶體位置和虛擬記憶體位置，讓每隻程式都擁有自己的記憶體空間。在執行程式時，則使用到 scheduler 決定每個程式之間的排程，達到多工的效果。

作業系統在開機時會先載入存在硬碟中的 boot loader 程式， boot loader 再將 kernel 載到記憶體 entry 的地方。%esp這隻指標是記憶體 stack 專門的指標，它會指在記憶體高位址的地方，並將 kernel 的 main 載到該位置，以預留多的記憶體空間給使用者程式。如此以來， kernel 就準備就緒，可以等著第一個使用者程式進入。 Kernel 執行第一隻使用者程式時會先呼叫 userinit ， userinit 再呼叫allocproc，在 process table 中創建一個位置給這個程序，等著 kernel 執行它。 接著， scheduler 在 process table 中尋找一個RUNNABLE 的程式，將其設成RUNNING並呼叫swtch，swtch再將所指的目標讀進暫存器裡，最後呼叫exec這個 system call，開始執行它。這就是整個xv6的開機流程及執行使用者程式的流程。

有作業系統在硬體和軟體之間溝通，才能讓這個設備達到它最大的功效。

項目三: 第四章 - Locking

Locking

在一個多CPU的硬體中，當有兩個或以上的CPU同時讀寫同一個記憶體位置，就很容易發生覆蓋資料或是讀取到未更新的資料等等問題，這種問題就被稱作一個 race 。為了解決這種問題，作業系統需要具備防止同一個記憶體被同時讀寫的機制，稱為lock。

Coding

Xv6實作了兩種lock， spin-lock 和 sleep-lock ，並使用acquire和release兩組函式來控制lock並保護其之間所處理的資料。如：













struct list* list = 0;
struct lock listlock;
void insert(int data)
{
    struct list* l;
    acquire(&listlock);
    l = malloc(sizeof(*l));
    l->data = data;
    l->next = list;
    list = l;
    release(&listlock);
}

acquire和release這兩組函式保證了其之間的程序只能被一隻CPU處理，保護了list資料的正確性，acquire用以取得lock的權限，而release則釋放lock。使用lock的機制有兩個：當一隻CPU在寫資料時，又有另一隻CPU也要讀寫他，就要使用lock；另外，當有一個非變數存在多個記憶體空間時，則需要用一隻lock來保護他們。

Xv6的 spin-lock 資料結構如下：








struct spinlock
{
    uint locked; // 是否有CPU占用此lock
    char* name; // lock的名字
    struct cpu* cpu; // 使用lock的CPU紀錄
    unit pcs[10];
};

Xv6的acquire實作如下：










void acquire(struct spinlock& lk)
{
    pushcli(); // 避免deadlock
    if(holding(lk)) // 判斷此CPU是否已持有lock
        panic(“acquire”);
    while (xchg(&lk->locked,1) != 0); // 一直迴圈到此CPU持有lock	
    lk -> cpu = cpu; // 紀錄哪隻CPU持有這隻lock
    getcallerpcs(&lk, lk->pcs); // 記錄進stack
}

其中特別的是他使用了x86中的xchg指令來保持lock的單一性，因為原生的C語言沒有保持程序處理單一性的指令。

Xv6的release實作如下：










void release(struct spinlock* lk)
{
    if (!holding(lk)) // 判斷此CPU是否已釋放lock
        panic (“release”);
    lk->pcs[0] = 0; // 釋放出lock
    lk->cpu = 0; // 清除CPU紀錄
    xchg (&lk->locked, 0); // 避免被重新排序\
    popcli();
}

流程圖

Deadlock

若一個系統中必須持有多個lock，則必須要保持他們的持有順序，否則恐怕會造成 deadlock 的情況。例如：程序A需要先持有lock A再持有lock B，而程序B需要先持有lock B再持有lock A，若程序A持有lock A時程序B也同時拿到了lock B，接下來就會出現兩個程序互拿不到對方的lock，又佔著自己的lock不放的情況，導致程序永遠無法執行下去，這種情況就稱作 deadlock 。
但是有些程序可能需要長時間佔著lock，而需要多個lock才能在效能間取得平衡，在xv6中則使用 sleep-lock 來解決此問題，用sleep和wakeup這兩個函式取得和釋放 sleep-lock 。

Xv6的 sleep-lock 資料結構如下：








struct sleeplock
{
    unit locked; // 判斷此lock是否被持有
    struct spinlock lk; // 被持有的lock
    char* name; // lock的名字
    int pid; // 該程序的pid
};

Xv6的sleep實作如下：























void sleep(void* channel, struct lock *lk)
{
    if (proc == 0)
        panic (“sleep”);
    if (lk == 0)
        panic (“sleep with lk”);
    if (lk != &ptable.lock)
    {
        acquire (&ptable.lock);
        release (lk);
    }
    // Go to sleep
    proc->chan = chan;
    proc->state = SLEEPING;
    sched();
    proc->chan = 0;
    if (lk != &ptable.lock) 
    {
        release (&ptable.lock);
        acquire (lk);
    }   
}

Xv6的wakeup實作如下：










void wakeup(void *chan)
{
    acquire (&ptable.lock);
    struct proc* p;
    for (p = ptable.proc; p > &ptable.proc[NPROC]; p++)
        if (p->state == SLEEPING && p-> chan == chan)
            p->state =RUNNABLE;
    release (&ptable.lock);
}

結論

使用lock保護資料的正確性絕對是必須的，但是多使用的話則會犧牲系統的平行性。在xv6中除了特定部分(如硬碟讀寫)，並無使用繁雜的lock，保持系統的效能。作業系統設計師必須非常明白使用lock的時機，才不會白白浪費硬體資源。
