項目二

作業系統提供了一個在軟體和硬體之間的介面

  • 將軟硬體之間的動作抽象化(system call)
  • 分配硬體資源給每一個程式達到多工的效果
  • 讓硬體的資源可以在不同的程式執行之間不會互相衝突

Processes and memory

一個 Process 分成兩個部分，程式的記憶體（指令、數據⋯⋯）與程式執行的狀態。xv6可以讓CPU在一段時間之內執行不同的程式，所以在等待的Process的狀態會被儲存起來，當要被執行時再將儲存的狀態放回暫存器中執行。

I/O and File descriptors

file descriptor 是一個整數，代表Process可以讀寫的對象，可以用read(fd, buf, n)從 file descriptor 讀出 n 個字元的資料。

為了讓CPU能在一段時間內執行多個程式且不衝突，每一個 process 的資訊會被紀錄起來，執行時以虛擬記憶體的方式透過 page table 對應到實際的記憶體上，這樣能確保每一個

Pipes

pipes 提供一個 kernel 的緩衝區，提供讀寫的 file descriptor 讓兩個 process 之間可以互相溝通。

File system

文件系統是一個樹狀的結構，每一個節點是文件或另外一個目錄。

chapter 1

一個作業系統的實作必須是：

  • 讓不同的程式分享電腦的資源
  • 讓不同的 process 之間獨立使用電腦資源，當某個程式執行錯誤時，其他的程式不能受到影響
  • 提供方式讓不同 process 之間不是完全分開的，他們能透過 pipes等方法互相溝通

Abstracting physical resources

作業系統不讓程式直接取得硬體的資源，而是透過抽象化的指令，讓程式透過作業系統的管理，去操作硬體資源。這也便於作業系統管理不同程式之間的切換，

User mode, kernel mode, and system calls

作業系統需要在應用程式之間有一道分明的界線，使得在應用程式出現錯誤的時候，處理例外並執行另一支程式，而不會影響到作業系統。
kernel 擁有特權可以對CPU進行 IO、記憶體存取、決定process 等動作，應用程式只能執行 user mode 的動作，當需要IO等動作時，必須透過 kernel 管理。

Process overview

透過將 process 抽象化，每個process佔用一小部分private的資源，避免CPU、file descriptor…的干擾。kernel需要謹慎的設計，因為只要有 process 的bug出現，就很容易導致整個系統崩潰。

項目三 - File system

buffer cache

  1. 記憶體中沒有重複的disk block
  2. 同一時間只有一個 kernel 使用某個 block，使用完會 call brelse。如果有其他的process要用卻還沒 release，就必須 wait
  3. cache 常用的 block 省時間，如果disk block沒有在cache裡，則會把 least recently used的block 放回disk中。

B_VALID buffer 已經從 disk 讀回資料
B_DIRTY buffer 中的資料已被修改過，需要寫回disk
B_BUSY buffer正在被使用，還沒有 release

// double linked list

  struct spinlock lock;
  struct buf buf[NBUF];

 // Linked list of all buffers, through prev/next.
 // head.next is most recently used.
 struct buf head;
} bcache;

binit() 初始化有 NBUF個 buf 的 double-linked list ，所有使用到的bufer cache的會從 bcache.head開始。

bread() 從 buffer 裡面用bget讀出一個 block。若這個 block 非 B_VALID 則需要iderw把 least recently used的block 放回disk中，並從硬碟讀出來。

bget() 在buffer中尋找 block 如果可用則將該block 設定為 B_BUSY 與回傳，如果正在被使用會 call sleep 直到被 release。如果沒有找到的話，會初始化一個buffer。若無就 panic("no buffer")

bwrite() 如果改動了 buffer 的資料，就必須設定 flag = B_DIRTY ，然後將 buffer 寫回 disk中

brelse() 當 process 使用完 buffer 之後，把 buffer 移動到 linked list 的第一個，清除 B_BUSY 然後通知那些正在sleep的 process。
linked list 的開頭代表 most recently used ，在 bget 的時候我們將 least recently used buffer 存回 disk 並重新拿來使用。

這樣的好處是當 locality 發揮效果的時候，我們可以用比較少的時間找到常用的 buffer。而因為搜尋需要走過全部的 buffer 所以直接可以處理 least recently used buffer。

logging

  1. 在系統崩潰後能夠恢復資料
  2. commit 代表完成動作，可以將幾個 system call group起來變成一組 commit
  3. 系統崩潰、重開機後，如果有 commit 的 log ，則把資料寫入 disk 否則就忽略

typical use of log in a system call looks like this:

begin_op();
//...
bp = bread(...);
bp->data[...] = ...;
log_write(bp);
//...
end_op();

重開機時 initlog 會在所有 process 之前執行，他會呼叫 recover_from_log() 然後執行類似 end_op()的事情，將 commit 過的 log 寫入 disk，然後把 log 清空。

begin_op() 等待 log system 直到沒有人正在 commit 且有足夠的 space。

log_write()把需要寫入的 block 記錄在memory中，而且把該 cache block 設為 B_DIRTY。

end_op() 如果 log.outstanding == 0 就會進行 commit()。
