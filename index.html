-----------------------項目二-------------------

作業系統功用

1.利用基本的機制來組合出強大的功能，如同樂高積木一般，由簡單的東西創造出複雜的功能。
2.對於多工的程式提供協調與調用，並且可以共享數據。
3.管理軟體與硬體之間的溝通，使得軟體不必多去煩惱這些問題。

殼層理念

在xv6中我們可以看到其使用殼層理念(shell)，使得程序(process)可以透過系統呼應(system call)來和內核(kernel)進行要求服務，其中服務則也包含了指令、數據、儲存空間等等。

而system call便是提供給用戶程式的系統接口，例如xv6便也提供了一系列的系統接口給予程式使用，而內核自身也會分配記憶體空間、分工、CPU等硬體資源的互動。

系統概要

程序(processes)和儲存(memory):
xv6在系統的內核中提供了分時的特性，利用不同的cpu之間的相互切換，可以實現多工和暫停的功能，當被執行的程式被暫停時，可以利用儲存暫存資料的方式，來切換至另外一組程式，等待再次返回執行至原本的程式時，便可以由暫存資料取得其執行中的值，並繼續該程式。

輸出入(I/O)和檔案調派(file descriptor):
檔案調派中利用一個整數來管理程式的讀寫，例如開啟文件、創建管道(pipe)等等，利用read與write來從文件讀取所需要的字節。

管道(pipe):
管道是僅限於內核中使用的的緩衝區，其作用便是將從檔案系統中提取出來的字節，以正確的形式給予程式，一般可以用於讀、寫的操作。

首次程序

雖說程序本身是指抽象的概念，使他可以享有其電腦的硬體資源，並且有利用概念的方式實現多工的CPU分配，而xv6利用了(page)來提供給各程序獨特的記憶體空間，並且將其管理至硬體資源上。

基本當系統開始運行時，系統會載入boot loader並且開始運行利用entry指令開始建立，並且在此時逐漸的建立表單的形式。

在等候entry成功建立記憶體表單之後便是main指令作用的時候，其會啟動並且初始化設備和子系統，運用userinit、mpmain、scheduler等等的指令之後便會進入第一個的system call。

-----------------------項目三-------------------

第四章: 鎖(Locking)

當xv6在多個處理器的裝置上運行時，便會有機會遇到資料衝突的情形，因多個處理器共享同一個記憶體時，其資料也會共享於同一個記憶體，但既使是單處理器的裝置也會遇到相同問題。

這時的xv6便會需要鎖(locking)的存在，其主要的功用便是利用鎖的互斥性，並且確保每個時間點中只有一個處理器在使用該資料結構，以此來避免覆蓋等會使資料損失的操作。

競爭狀態(Race Conditions)

由下面的程式碼我們可以預期到，有兩個處理器會在相同的時間點向硬體資源發出link list的請求，在以下的情況下唯一可以決定程序的寫入順序的只剩下處理器的執行時間跟存取先後，這是十分冒險且不可靠的情況，以下的程式碼中便有該問題：

struct list{
    int data;
    struct list *next;
};

struct list *list = 0;

void
insert(int data)
{
    struct list *l;
    l = malloc(sizeof *l);
    l->data = data;
    l->next = list;
    list = l;
}

該情況則是標準的競爭狀態，最保險的方法便是利用鎖的互斥性來避免在相同的時間下發生同時的資料存取，略在程式碼中加入獲得鎖(acquire)以及解開鎖(release)的動作便可以以安全且可靠的方式避免競爭：

struct list *list = 0;
struct lock listlock;

void
insert(int data)
{
    struct list *l;
       acquire(&listlock);
    l = malloc(sizeof *l);
    l->data = data;
    l->next = list;
    list = l;
       release(&listlock);
}

由此我們可以藉由鎖的特性保存資料的不變量，而在這次程式碼的list便是不變量，對於程序的操作而言，正確與否的操作則與不變量的是否為真有著不小的關係，儘管可以短暫的破壞不變量，但也必須在下次使用前還原狀態。

鎖(locking)的構造

但對於xv6來說鎖也是有分成：spinlock 和sleeplock，以下便是探倒spinlock的問題：

void
acquire(struct spinlock *lk)
{
    for(;;) {
        if(!lk->locked) {
            lk->locked = 1;
            break;
        }
    }
}

這段程式碼中簡單的實現了鎖的功能，但同時也有問題存在於這段程式碼中，現代的處理器中多半是多工與多核心的，而當有一個以上的處理器同時呼叫鎖時，便會發生問題

為此我們需要使用xchg這個特殊指令，其可以利用atomic step的特性，將記憶體中的值或字交換，就可以避免該狀況

在鎖的功能中可以分為大致幾個指令：

獲得鎖(Acquire):
使得鎖的持有者可以修改記憶體的資料，並且記憶持有鎖的處理器，且可以在發生死鎖時偵測問題。

釋放鎖(Release):
將獲得的鎖釋放出去，避免發生由同一程序永遠拿著鎖的情況，避免死鎖的發生。

xchg:
利用但次的atomic step來確保處理器無法在同時獲得鎖，由此可以確保鎖的功用。

鎖的難處

鎖雖說可以保護資料的結構，但同時也有存在著缺點：
1.在使用鎖的時候會降低並發處理，導致效率降低
2.決定該處理器所需要的鎖，以及要鎖上何種資料
3.能避免使用鎖的情況只使用於單處理器的裝置

而其中也有類似內核主鎖（giant kernel lock）的方式，但會犧牲大量的並發性，卻可以安全的保護內核中的資料結構，由殼層進出入內核都需要該鎖，這也使得只有單一處理器可以運用內核。

死鎖與鎖上順序

在同一段的程式中如果需要呼叫多個鎖，但如果其鎖上的順序是相同的，便會出現死鎖的風險。
而在xv6系統中同一段的程序則可能呼叫兩個鎖。

中斷程序

xv6利用鎖來防止中斷程序和非中斷中的程序使用同數據來導致資料干擾，但儘管在單個處理器上，中斷也是有可能發生，所以為了避免這個情況，我們必須確保處理器不得在中斷發生時還持有鎖。

為此我們必須秉持幾個重點：
1.必須要使用pushcli與popcli來處理中斷
2.acquire必須在獲得鎖之前xchg調用pushcli，在兩者顛倒的情況下則會發生系統死鎖。
3.在中斷中的時候，不准持有任何鎖

記憶體混亂

在前面的狀況中我們都預想了處理器會依照程序的順序來執行，但對於現今的處理器來說，多半會利用指令亂序來提高性能，處理器便可以與其他指令交疊，來避免延誤。
但這種行為回讓有問題的資料暴露到軟體之中，這也更加容易發生錯誤，為了避免該混亂的出現，xchg便可以使用。
