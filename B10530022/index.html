項目二

  • xv6使用了傳統kernel的概念
  • 每個運行中的program(稱為process)擁有instructions, data, and a stack的memory.
  • 一個xv6 process包含了 user-space memory 和 只有kernel可見的state

Kernal 有 hard privileges 實現硬體保護機制

system call 在OS介面上呼叫的過程

system call 使process可以在user空間和kernel空間互相作用。

  • fork() 創建一個新的process(又稱 child process)
    -child process 由 parent process 而來
    -在parent中，fork 回傳 child’s pid
    -在child中，fork 回傳 0
  • exec 可從某個文件中讀取memory image,並替換到memory空間中
    -包含兩個參數，一個是可執行文件名稱，另一個是字串列
  • sbrk(n) 可用來增加n個字節

file descriptor

是一個整數，被用來表示可以讀寫的 kernel-managed object

  • read(fd, buf, n) 從 fd 中讀取最多 n 個字節到 buf 中
  • write(fd, buf, n) 寫 buf 中的 n 個字節到fd裡
  • close(fd) 可釋出 file descriptor
    -以後可以被 open, pipe, dup 重新呼叫
    -一個新被釋出的 file descriptor 會是當前最小的未被使用 file descriptor

I/O 重定向

可以由 file descriptor 和 fork() 交互作用實現

  • parent 和 child 之間共享文件偏移量(file offset)
  • dup(fd) 可以複製一個存在的 file descriptor fd
  • 指向同一個輸出/入對象的 file descriptor 共享文件偏移量

pipe

int p[2];
char *argv[2];
argv[0] = "wc";
argv[1] = 0;
pipe(p);
if(fork() == 0) {
    close(0);
    dup(p[0]);
    close(p[0]);
    close(p[1]);
    exec("/bin/wc", argv);
} else {
    write(p[1], "hello world\n", 12);
    close(p[0]);
    close(p[1]);
}
  • 執行 wc 之前要關閉child process的寫入端，避免如果 wc pipe 指向一個pipe的寫入端，wc 就讀不到 eof 了

    -inode號(ino) 相同的兩個不同文件名的文件，指的是同個文件

monolithic kernel(單內核)

整個 OS 都在 kernel 中，所有的 system calls 都在 kernel 中實現。

優點：

  • 已完全硬體權限運行，對設計人員來說很方便
  • OS 中個不同部分很容易合作

缺點：

  • 不同部分的接口很複雜
  • kernel 模式下發生錯誤，會導致kernel 失敗 ，使得應用程式也失敗

microkernel

在 user mode 下執行大部分的 OS

process overview

  • xv6 使用 struct proc 來維護 process的狀態
  • 用 p->xxx 來表示 proc 的元素

Code: the first address space

  • 初始化，從磁碟中載入 boot loader並執行
  • boot load 將xv6 kernel 載入物理地址 0x100000
    (0xa0000 到 0x100000 : I/O)
  • entrypgdir 的物理地址载入到控制暫存器 %cr3 中
  • stack pointer %esp 指向被作為 stack 的 memory
  • entry 間接跳轉到高地址 main

Code: creating the first process

  • 呼叫 userinit 創建第一個 process , userinit 會再呼叫 allocproc
  • allocproc 在 process table中分配一個槽(slot) (a struct proc) ，並初始化狀態
  • allocproc 會在 proc table 中找到標記為 UNUSED 的槽，必把狀態改為 EMBRYO
  • 將 p->context->eip 指向 forkret ，讓 kernel thread 從 forkret 2533 的開頭開始執行
  • context switch code 將 stack pointer指向 p->context 的结尾
  • userinit 呼叫 setupkvm 為 process 創建一個 page table
  • userinit 將 p->state 設為 RUNNABLE

Code: Running the first process

  • mpmain 呼叫 scheduler 執行 process
  • scheduler）找到一個 p->state 為 RUNNABLE 的 process ““initproc””，設per-cpu 的變量為 initproc
  • switchuvm 設好工作狀態段 SEG_TSS
  • scheduler 把 p->state 設為 RUNNING
  • swtch 會儲存當前的暫存器，然後把storage (proc->context) 載到 x86 的硬體暫存器中
  • scheduler 會叫 swtch 把當前的硬體暫存器儲存在per-cpu 的 storage (cpu->scheduler) 中
  • allocproc 透過把 initproc 的 p->context->eip 設為 forkret ，使得 ret 執行 forkret

The first system call: exec

  • initcode.S 一開始會把暫存器 argv，init，0 三個值推到 stack ，然後把 %eax 設為 SYS_exec 在執行 int T_SYSCALL
  • 執行成功，exec 不會回傳，並執行 $init 的 process
  • 執行失敗，exec 回傳，initcode 不斷呼叫一個不會回傳的系統，叫他呼叫exit

項目三-Locking

建立合作機制防止 CPU 互相干擾

防止中斷程序與非中斷程序互相干擾

避免競爭條件

locking 的一些名詞

  • 不變量 一個數據狀態，從操作開始前與操作結束需一樣，過程中則無限制
  • 原子操作 並發編程中的原子操作是完全獨立於任何其他進程運行的程序操作
  • 死鎖 多個程序等待同一個鎖，導致程序永遠處在等待
  • 競爭 使用到同一個賦值，使後來程序的賦值蓋掉前一個
  • 互斥 執行程序時一次只有一個CPU可以運行

Lock 的順序 代碼運行時每次使用 Lock 的順序需保持一樣，避免死鎖

  • 優點
    -較容易避免執行並發計算時產生的競爭
  • 缺點
    -需額外的工具來確認競爭條件避免遺漏掉需要保護的不定量
    -大部分需要使用原子操作，進而佔據更多緩存

操作流程

資料結構

initlock(struct spinlock *lk, char *name)
{
  lk->name = name;
  lk->locked = 0;
  lk->cpu = 0;
}
  • 獲取 lock
  • 迴圈執行直到獲得lock
  • 有可能一直持有 lock ,那麼其他的 CPU 會為了獲取lock浪費時間
void acquire(struct spinlock *)	
{
  pushcli(); //為了避免死鎖，禁止使用中斷
  if(holding(lk))
    panic("acquire");

  //xchg 是原子操作(獨立進行)
  while(xchg(&lk->locked, 1) != 0)
    ;

  //紀錄有關lock的訊息
  lk->cpu = cpu;
  getcallerpcs(&lk, lk->pcs);
}
  • 釋出lock
void release(struct spinlock *lk)
{
  if(!holding(lk))
    panic("release");

  lk->pcs[0] = 0;
  lk->cpu = 0;

  //xchg序列化，因此在發布之前的讀取不會在它之後重新排序
  xchg(&lk->locked, 0);

  popcli();
}
  • 按照 %ebp 鏈紀錄 pcs[] 目前呼叫的 stack
void getcallerpcs(void *v, uint pcs[])
{
  uint *ebp;
  int i;
  
  ebp = (uint*)v - 2;
  for(i = 0; i < 10; i++){
    if(ebp == 0 || ebp < (uint*)KERNBASE || ebp == (uint*)0xffffffff)
      break;
    pcs[i] = ebp[1];     //存入 %eip
    ebp = (uint*)ebp[0]; //存入 %ebp
  }
  for(; i < 10; i++)
    pcs[i] = 0;
}
  • 檢查此 CPU 有沒有持有 lock
int holding(struct spinlock *)
{
  return lock->locked && lock->cpu == cpu;
}

void panic(char * s)	
{
  int i;
  uint pcs[10];
  
  cli();
  cons.locking = 0;
  cprintf("cpu%d: panic: ", cpu->id);
  cprintf(s);
  cprintf("\n");
  getcallerpcs(&s, pcs);
  for(i=0; i<10; i++)
    cprintf(" %p", pcs[i]);
  panicked = 1; //凍結其他CPU
  for(;;)
    ;
}
  • 需要兩個popcli 來消除兩個 pushcli
  • 如果中斷關閉，popcli 和 pushcli 也會關閉
void pushcli()	
{
  int eflags;
  
  eflags = readeflags(); //取得目前狀態
  cli();
  if(cpu->ncli++ == 0) //若為0則存入目前狀態
    cpu->intena = eflags & FL_IF;
}

void popcli()	
{
  if(readeflags()&FL_IF)
    panic("popcli - interruptible");
  if(--cpu->ncli < 0)
    panic("popcli");
  if(cpu->ncli == 0 && cpu->intena)
    sti();
}
