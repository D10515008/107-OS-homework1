項目二:

第零章

operating system的工作
(1) 將computer的資源在多個program間共享，並且給program提供一系列比hardware本身更有用的服務。
(2) 管理並抽象化底層的硬體，舉例來說，一個文字處理軟件不用去關心自己使用的是哪種disk。
(3)hardware共享使得multiple programs可以同時運行(多工)
(4)作業系統提供控制方式給program去interact交流，所以能夠共享data共同工作。process通過system call使用內核服務。system call會進入內核，讓內核執行服務然後返回。所以process總是在用戶空間和內核空間之間交替運行。

第一章

xv6是用在Intel80386(x86)與其之後的處理器上，而地址是從0x80100000開始，每個process都有用戶棧和內核(kernel)棧。當process運行用戶指令時，只有用戶棧被使用，而內核棧則是空的。然而當process（通過系統調用或中斷）進入kernel時，內核代碼就在process的內核棧中執行；process處於kernel中時，其用戶棧仍然保存著數據，只是暫時處於不活躍狀態。process的線程交替地使用著用戶棧和內核棧。必須要注意的是內核棧是用戶代碼無法使用的，這樣即使一個進程破壞了自己的用戶棧，內核也能夠繼續保持運行。當我們的PC開機時，它會初始化自己然後從磁盤中載入 boot loader 到內存並運行起來，然後，boot loader 把 xv6 內核從磁盤中載入並從 entry（1040）開始運行。

項目三:

Traps, interrupts, and drivers

前言:

當一個 process 在執行的時候，CPU 會做一個normal processor loop：

  1. 讀取 instruction
  2. program counter 更新
  3. 執行 instruction

user program 會使得某些 事件使得program轉回kernel而非執行下一行 instruction，處理這些 event 會遇到主要這三種問題：

  1. kernel 必須讓 processor 從 user mode 轉移至 kernel mode
  2. kernel 和 device 必須對應到 parallel activity
  3. kernel 必須了解 device 的 interface

Systems calls, exceptions, and interrupts:

有三種情況控制權限必須從user mode轉換回到kernel mode
1.system call:當使用者想使用operating system的服務
2.exception:當使用者做了一個不合法的行為時
3.interrupt:當device發出訊號想引起OS的注意

當中斷產生時，基本的動作如下:
1.中斷normal processor loop並開始執行名為interrupt handler
2.儲存被暫停的processor的register(以便障礙排除後回到user mode繼續執行。processor 需要從 user mode 轉移到 kernel mode 再回到 user mode，是在轉入轉出 interrupt handler 時需要面對的問題。)

X86 protection:

在x86之中，interrupt descriptor table(IDT)有256個entry
程式藉由int指令來產生interrupt去呼叫system call，而int會改變權限階級，若int沒有要求改變權限階級，則x86不會儲存%ss和%esp

x86規定了中斷、異常、系統調用的規範，在發生以上情況時，硬體能夠區分上述規定的256種觸發中斷的原因並通過尋找在記憶體中保存的中斷向量表來得到中斷處理常式的位址，然後將控制權交給中斷處理常式，這是x86架構下發生中斷時硬體完成的操作。

Code: The first system call:

void tvinit(void)
{
int i;
for(i = 0; i < 256; i++)
SETGATE(idt[i], 0, SEG_KCODE<<3, vectors[i], 0);
SETGATE(idt[T_SYSCALL], 1, SEG_KCODE<<3, vectors[T_SYSCALL], DPL_USER);
initlock(&tickslock, “time”);
}

• tvinit 建立一個 256 個 entry 的 table idt
• interrupt i 存在 vector[i]之中
• tvinit 處理 TSYSCALL，user system 稱它為 trap，並定義 gate 是一個 trap 的 type，trap gate不會清除 IF flag，而且在 system 呼叫 handler 時，容許其它的 interrupt
• kernel 也有設定一個 system call gate privilege 給 DPL_USER，DPL_USER 允許 user program 產生一個 int instruction 的 trap，不過xv6 則否，若 xv6 常是如此，會有 general protection excrption 出現

trap:

void trap(struct trapframe *tf)
{
// 由tf->trapno決定用處
if (tf−>trapno == T_SYSCALL) //呼叫syscall
{
if (myproc()−>killed)
exit();
myproc()−>tf = tf;
syscall();
if (myproc()−>killed)
exit();
return;
}
// 找尋hardware interrupts
switch (tf−>trapno)
{
case T_IRQ0 + IRQ_TIMER:
if (cpuid() == 0)
{
acquire(&tickslock);
ticks++;
wakeup(&ticks);
release(&tickslock);
}
lapiceoi();
break;
case T_IRQ0 + IRQ_IDE:
ideintr();
lapiceoi();
break;
case T_IRQ0 + IRQ_IDE + 1:
break;
case T_IRQ0 + IRQ_KBD:
kbdintr();
lapiceoi();
break;
case T_IRQ0 + IRQ_COM1:
uartintr();
lapiceoi();
break;
case T_IRQ0 + 7:
case T_IRQ0 + IRQ_SPURIOUS:
cprintf(“cpu%d: spurious interrupt at %x:%x\n”,
cpuid(), tf−>cs, tf−>eip);
lapiceoi();
break;
//並不是system call或 hardware interrupts
default:
if (myproc() == 0 || (tf−>cs & 3) == 0)
{
cprintf(“unexpected trap %d from cpu %d eip %x (cr2=0x%x)\n”,
tf−>trapno, cpuid(), tf−>eip, rcr2());
panic(“trap”);
}
cprintf("pid %d %s: trap %d err %d on cpu %d "
“eip 0x%x addr 0x%x−−kill proc\n”,
myproc()−>pid, myproc()−>name, tf−>trapno,
tf−>err, cpuid(), tf−>eip, rcr2());
myproc()->killed = 1;
}
// proc()->killed清除user的 process
// 強迫process終止
if (myproc() && myproc()−>killed && (tf−>cs & 3) == DPL_USER)
exit();

if (myproc() && myproc()−>state == RUNNING && tf−>trapno == T_IRQ0 + IRQ_TIMER)
yield();

// 檢查process是否終止
if (myproc() && myproc()−>killed && (tf−>cs & 3) == DPL_USER)
exit();
}

Code: System calls:

void syscall(void)
{
int num;
struct proc *curproc = myproc();
num = curproc−>tf−>eax;
if(num > 0 && num < NELEM(syscalls) && syscalls[num])
{
curproc−>tf−>eax = syscalls[num]();
}
else
{
cprintf("%d %s: unknown sys call %d\n",
curproc−>pid, curproc−>name, num);
curproc−>tf−>eax = −1;
}
}
在 system call 裏頭，trap 會去呼叫 syscall，syscall 從有存取 %eax 的 trap frame 載入 system call number，再去對應到 system call table
由%eax記錄 system call function 的回傳值
system call 一般會return一個負數來表示false的情形，正數表示true；如果 system call number 不合法，syscall 會印出 error 並且回傳 -1
