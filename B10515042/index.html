第零章

作業系統存在的原因在於，如果只有一隻程式再跑那當然沒什麼問題，但是當多隻程式在跑時，就資源上必須要共享，要怎麼共享呢，則需藉由作業系統來分配。這一章節著重在介紹一個作業系統最基本因有的功能：process, memory, file descriptors, pipes, file system。

第一章

一個作業系統最主要的三個精神：multiplexing, isolation, interaction。

其中提到，為了要方便程式撰寫，所以在每隻程式執行時，會給他一個完整的虛擬記憶體，此虛擬記憶體裡就像是一個hash table會對應到實體的記憶體或是disk等儲存空間。這個想法最主要是可以達成三個精神裡的isolation，因為這不會使兩個不同program的process用到一樣的記憶體位址，造成程式上的錯誤。

在kernel organization這章裡提到，一個最簡單管理硬體及資源的方法就是將所有作業系統的範圍全部塞到kernel，稱monolithic kernel，但是這會遇到一個問題，因為作業系統非常的複雜，所以當一個程式出錯時，就會導致kernel爆開，而爆開的結果就是重新開機，所以為了避免這個狀況發生，有些作業系統的設計方式會將部分的功能放置在外圍，減輕kernel的壓力，稱作microkernel。

Locking

最主要我們要確保的是一個data在一個時間點只能被一個CPU給使用或鎖住資源。

Race condition

這個情況會發生的原因在於某個data structure需要藉由一系列的operation去維護他也就是所說的invariant，所以我們必須要有個lock來保護這個程序避免其他的干擾，舉個例子來說：disk driver因為是共用的所以他會維護一個linked list，而其程式碼如下左，在第13～16所做的程序為新增一個list並且放入data以及連接下個list接著將l這個指標更改指向到現在這個新的，一切就如同正常的linked list這個data structure的運作模式，但是若有兩個程序同時執行道第15行時，會發現有兩個的linked同時指向list但是要注意的是list並沒有更新，而進行到第16行時，雖然是更新了，不過list會指向的是第二個程序的，所以這就是race condition。而更改方式為右邊這張圖，他首先先在12行lock住這個程序不給其他CPU執行，所以就可以避免這一情況。

Deadlock and lock ordering

deadlock會產生的情況就是，當CPUA lock 住A data, CPUB lock住B data時，接著CPUA 要求B data 而CPUA 要求 A data，則這個時候就會陷入無限等待，因為兩個CPU都無法去lock住新的data。主要的方法就是確保每個lock的順序都是固定的，也就是當CPU要lock data的順序是 A->B->C，那這樣當CPUB要lock B時，他就必須等到A沒被lock才能lock，那這樣就可以減少deadlock的機會。就像下面這程式碼的第5~10行的檢查順序，這樣就不會產生deadlock了，就可以去lock資源了（第12行）。

Interrupt handler

當interrupt發生時，所有的process都要停止動作去做呼叫interrupt的function，但是這時候若是這個function需要被停止動作的function的data時就會產生deadlock，無法執行下去，所以為了避免此一行為，再每個process在lock住data之前時，都必須要呼叫pushcli()來關掉interrupt，而結束時再開啟interrupt。左邊這張圖在1660這行將interrupt關掉並且接著的幾行是拿來計數，看有lock住幾個data；而右邊這張圖，在1671,1672是在每free一個data少一個可能需要關閉interrupt的機會，在1673,1674是判斷若計數為0則開啟interrupt。

Instruction and memory ordering

因為processor或compiler欲增加執行速度，所以會將不會互相影響的process放在不同的core執行，但是若是遇到某些不允許這樣的情況時，就會造成程式執行錯誤，為了避免這樣在acquire與release會使用__sync_synchronize()來確保可以按照順序執行。

Sleep lock

在我們使用file system時因為要等很久的時間，為了節省時間，xv6會給其他的thread執行，並且將這個file lock的非常的穩，我們稱為sleep_locks。但是這會有一個問題，如果其他的tread需要被sleep_locks的thread的資源時，會造成另一個deadlock，因為被lock住了且此一thread已經關掉了interrupt所以也阻止了被sleep_locks的thread執行。為了避面這樣，在sleep_locks的thread會執行acquiresleep()，他會釋放他所使用的資源，來讓其他人使用。下圖為sleeplock的資料結構。
