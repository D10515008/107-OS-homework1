    第零章:

    一個xv6進程包含用戶空間內存的指令，數據，堆疊的內存空間，以及僅對kernel可見的進程狀態。Kernel會將每一個進程與一個進程識別碼(pid)連結。

    xv6的shell調用fork, wait和exec等system call來為用戶執行程序：
    　　　1. 讀入用戶命令
    　　　2. 使用fork產生shell進程的副本
    　　　3. 父進程調用wait，子進程執行命令，調用exec。
    　　　4.
    　其中fork建立一個新的(子)進程，父子擁有相同內容但不同的內存和寄存器。
    　wait會返回當前進程中子進程的pid，函數將會等待直到有子進程結束。
    　exec函數將特定(通常可執行並符合格式)文件的內容取代至調用它的進程的內存。

    進程可以讀/寫的由kernel管理的文件由文件描述符(整數)代表，對應的可能是文件、pipe或是裝置的抽象。Shell以此實現I/O的重新指向和pipelines。

    system call中的pipe()用於建立管道(kernel緩衝)，以文件描述符的形式提供給進程。提供一個進程間交互溝通的方式。

    如此作業系統可以達成：
    * 抽象化並管理底層硬體
    * 讓多個程式共享硬體
    * 提供程式交互作用的方式

    第一章:

    作業系統的一個重要需求是支援多個作業的同時進行，而分時(time-share)需要滿足三項特性：multiplexing、isolation和interaction。

    xv6中isolation的單位為進程，其抽象化提供程式私有的對應內存系統address space，產生程式在獨立系統中運作的錯覺，以防止對其他指令乃至於kernek的破壞和觀察。

    進程擁有各自的線程來執行進程的指令。線程可以被暫停，於稍後再恢復。進程之間切換實際上就是暫停一者而恢復另一者的運行。線程的大多數狀態則保存在線程的堆疊上。

    p->state指出進程為新建、準備運行、運行中，等待I/O或退出中。
    p->pgdir保存符合x86硬件的格式的進程頁表。

  • 項目三:

    第五章: 調度

      1. 進程數量大多時候多過處理器數量，因此需要分享處理器
  2. 而分時處理器則需要特定的計畫
  3. 理想狀況下分時的作業不被用戶進程察覺

    • 多路復用
      在xv6中，以sleep/wakeup機制在一個進程等待(裝置、pipe I/O、子進程結束等)時，使其進入睡眠，然後執行另一個進程。這樣的多路復用機制提供了進程獨占處理器的假象，類似於使用內存分配器和硬件頁表提供獨占內存假象的方式。

      實現多路復用有幾個難點:

      1. 如何切換進程 - 上下文切換機制
      2. 如何讓上下文切換透明化 - 定時器中斷處理程序
      3. 可能出現多個CPU同時切換進程的情況，則須使用locking(ch4)來避免競爭
      4. 進程退出時必須釋放佔用的內存與資源，但進程無法釋放本身在的資源（如其kernel stack）

      xv6須為進程提供互相協作的方法，但為避免等待中的進程消耗CPU資源，則讓它們暫時睡眠，並允許其他進程重新喚醒他們。

    • 上下文切換

      xv6在低層次中實現了用戶程序的切換：

      • 用戶空->kernel空間(kernel線程->CPU的調節器線程->的kernel線程)->用戶空間

      xv6不會直接在戶態進程間切換。每個進程都有自己的kernel堆疊以及寄存器集合，而CPU都有單獨的調度器線程，如此調度是在調度器線程而非進程的kernel線程中。切換涉及到了保存舊線程的CPU寄存器，恢復新線程之前的寄存器。
      swtch函數不知道線程的存在，而只單純儲存/復原寄存器(即上下文)。當進程要放棄使用CPU時，其kernel線程呼叫swtch儲存自己的內容並返回至調度器的內容。上下文以指向kernel堆疊中structure的struct context* 代表：

      • void swtch(struct context **old, struct context *new)

      push舊CPU寄存器至堆疊上並把堆疊的指標存於*old；將new複製到%esp、pop上一個儲存的寄存器後返回。

    • 調度

      調度器(scheduler)執行一個loop(如流程圖右半)：
      1. 在process table找尋p->state==RUNNABLE的進程
      2. 切換至此進程、執行直到停止
      3. 迴圈至1
        迴圈中的步驟2，調度器會先設定per-CPU當前進程變量proc、以switchuvm切換進程頁表、標示此進程為RUNNING，最後呼叫swtch開始執行進程。
        當進程要讓出CPU(流程圖左半)時：
      4. 必須要獲得進程表的鎖ptable.lock、釋放此進程持有的其他鎖
      5. 修改進程狀態proc->state
      6. 調用sched
      7. sched調用swtch
      8. sleep&exit
        調度代碼需要維護進程的一系列狀態、保證狀態變化時必須持有ptable.lock。

    • 睡眠與喚醒
      sleep和wakeup為幫助進程間互相特意互動的虛擬化，讓一進程可以睡眠以等待另一進程，反之另一進程可以在事件發生時喚醒睡眠的進程。

      以生產者/消費者的佇列為例－類似IDE驅動用以同步處理器和裝置驅動的佇列：

      • send不斷循環直到佇列為空(ptr==0)，然後將指標放入佇列
      • recv循環至，將指標放入佇列為非空後，取出指標
      • 不同的進程運行時，send和recv會同時修改q->ptr，但send只於佇列空時寫入，而recv於非空時取出，互不干擾。
      





















      struct q {
    void *ptr;
};

void*
send(struct q *q, void *p)
{
    while(q->ptr != 0)
        ;
    q->ptr = p;
}

void*
recv(struct q *q)
{
    void *p;

    while((p = q->ptr) == 0)
        ;
    q->ptr = 0;
    return p;
}

      上述實現方式代價高昂。若發送者很少發送，那麼接受者會消耗大量在等待。

      加入sleep/wakeup的機制：

      • sleep(chan)使抽象值chan休眠，稱之為等待佇列
      • wakeup(chan)則喚醒所有在chan中休眠的進程，使它們的sleep呼叫返回。若沒有休眠的進程，wakeup不回產生任何作用
      

















      void*
send(struct q *q, void *p)
{
    while(q->ptr != 0)
        ;
    q->ptr = p;
    wakeup(q);    /*喚醒recv*/
}

void*
recv(struct q *q)
{
    void *p;
    while((p = q->ptr) == 0)
      sleep(q);
   q->ptr = 0;
   return p;
}

      然而這樣的實作可能會因為沒有維持好固定狀態而遭遇"lost wake-up"問題。

      解決方式： 加入鎖(ch3)

      • 要呼叫sleep的進程必須持有鎖，反之要呼叫wakeup亦然
      • 為避免"死鎖"問題的發生，呼叫進程需要將鎖作為參數提供給sleep，讓函式能在將呼叫進程標示為休眠中後將鎖釋放

      修正後的代碼如下：

      


























      struct q {
    struct spinlock lock;
    void *ptr;
};

void *
send(struct q *q, void *p)
{
    acquire(&q->lock);    /*取得鎖*/
    while(q->ptr != 0)
        ;
    q->ptr = p;
    wakeup(q);
    release(&q->lock);    /*釋放鎖*/
}

void*
recv(struct q *q)
{
    void *p;
    acquire(&q->lock);
    while((p = q->ptr) == 0)
        sleep(q, &q->lock);    /*sleep的參數中提供鎖*/
    q->ptr = 0;
    release(&q->lock;
    return p;
}

      總體來說：

      • sleep將當前進程轉為SLEEPING並呼叫sched以釋放CPU
      • wakeup尋找一個SLEEPING的進程並把它標記為RUNNABLE

    • 管道
      第0章介紹管道的接口：

      1. 從管道的一端寫入數據字節
      2. 數據被拷貝到內核緩衝區中
      3. 可自管道的另一端讀取數據

      Pipewrite和piperead的實現：
      管道由結構struct pipi實現。其中包含一個鎖lock和內存緩衝的、nread和nwrite兩個field紀錄從緩衝區讀出和寫入的字節數。


      如流程圖中所示：

      1. Pipewrite請求獲得管道的鎖，以保護計數器，數據以及相關不變量
      2. Piperead請求獲得鎖，無法獲得，於是停在acquire等待鎖的釋放。期間pipewrite將loop中依次寫入的addr[0]至addr[n-1]添加到管道中
      3. Loop中，緩衝區可能被寫滿，這時pipewrite呼叫wakeup通知睡眠中的讀者緩衝區中有數據可讀
      4. &p->nwrite隊列中睡眠的讀者自緩衝區中讀出數據
      5. Sleep讓pipewrite的進程進入睡眠並釋放p->lock
      6. Piperead獲得該鎖然後開始執行，並檢查到p->nread != p->nwrite
      7. 跳出loop，自管道中拷貝數據、增加nread的計數
      8. Piperead呼叫wakeup喚醒睡眠中的piperead
      9. 返回至呼叫者中

    • Wait、exit和kill
      Wait：

      1. 要求獲得ptable.lock
      2. 查看進程表中是否有子進程，如果存在子進程，但無任一已經退出，則呼叫sleep等待任何退出者。
      3. 不斷循環

      Exit(讓一個進程可自我終結)：

      1. 要求獲得ptable.lock
      2. 喚醒前進程的父進程
      3. 調用sched讓出CPU

      Kill(讓一個進程可終結其他進程)：

      1. 將要終結的進程設為p->killed
        • 如果這個進程在休眠中則將其喚醒
        • 如果要終結的進程是在用戶空間運行，進程會很快的通過系統調用或者中斷進入內核
      2. 當要終結的進程離開kernel後，若p->killed，trap會呼叫exit來終結進程
    • 現實情況
      • xv6的調度算法十分簡單－讓每個進程輪流執行－稱作輪轉法(round robin)。真正的操作系統使用更複雜的算法，例如讓進程各有優先級，優先處理高優先級的可運行進程。
      • Sleep和wakeup是普通但有效的同步方法。無論同步方法，要解決的第一個問題是章節中(ch5)提到的"lost wake-up"問題。
      • 搜尋整個進程表來得到對應chan的進程是非常低效的。更好的辦法是以包含休眠進程的list資料結構來取代sleep和wakeup中的chan。此類機制都的共通點是：睡眠狀態以能在睡眠期間自行脫落的鎖來保護。
