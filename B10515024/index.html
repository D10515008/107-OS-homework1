Processes and memory

  • fork()
    • create A new child process with the same memory content from parent process.
    • return children’s pid.(in children return 0)

  • wait()
    • all returns the pid of anexited child of the current process; if none of the caller’s children has exited,waitwaits for one to do so.
  • exec()
    • it needs program’s location and following argument(s)
    • The exec system call replaces the calling process’s memory with a new memoryimage loaded from a file stored in the file system.
    • When exec succeeds, it does not return to the calling program; instead,the instructions loaded from the file start executing atthe entry point declared in the ELF header.
  • exit()
    • The exit system call causes the calling process to stop executing and to release resources.
  • getcmd()
    • get users’ command from prompt(loop)
  • sbrk(n)
    • A process that needs more memory at run-time (perhapsformalloc) can callsbrk(n)to grow it’s data memory by n bytes.

I/O and File descriptors

  • A file descriptoris a small integer representing a kernel-managed object that a process may read from or write to.
  • A process may obtain a file descriptor by opening a file, directory, or device, or by creating a pipe, ollr by duplicating an existing descriptor.
  • file descriptor:
    file descriptor meaning
    0 standard input
    1 standard output
    2 standard error
  • write(fd, buf, n) / read(fd, buf, n)
    • writes nbytes from buf to the file descriptor fd and returns the number of bytes written.
    • read(fd, buf, n) reads at most n bytes from the file descriptor fd, copies them into buf, and returns the number of bytes read.

Pipe

  • small kernel buffer exposed to processes as a pair of file descriptors
    • reading / writing
    • echo helloworld | wc
    • if we don’t have pipe:
      echo hello world >/tmp/zyz; wc </tmp/xyz
  • pipe’s advantage
    1. pipes automatically clean themselves up
    2. pipes can pass arbitrarily l ong streams of data, while file redirectio n requires enough free space on disk to store all the data
    3. pipes allow for parallel execution of pipeline stages, while the file approach requires the first program to finish before the second starts
    4. if you are implementing interprocess communication, pipes’ blocking reads and writes are more efficient than the non-blocking semantics of files
  • mknod
    • make a file without contents
  • fstat
    • retrieves information about the object a file descriptor refers to
  • link
    • creates another file system name referring to the same inode as an existing file
    • we can access file with names “a” and “b”
  • unlink
    • unlink “a”, but still can be accessed with “b”
    • when link number is 0 then, the file will be wipe out from disk

Process overview

  • isolating kernel and user
    • user can’t r/w kernels’ memory
  • In order to enhence isolation between kenel and user, because of buggy application or irrational usage of kernel.
  • struc proc

    • Each process has a thread of execution

      • it can be suspended/resumed

    • Each process has two stacks

      • kernel stack (p->kstack)
        • process is executing on kernel stack
      • user stack (p->ustack)
        • process is executing on user stack and kernel stack is empty
      • when process switch from user to kernel
        • user stack is frozened until process return to user stack

        When a process makes a system call , the processor switches to the kernel stack, raises the hardware privilege level , and starts executing the kernel instructions that implement the system call. When the system call completes, the kernel returns to user spac e: the hardware lowers its privilege level, switches back to the user stack, and resumes executing use r instructions just after the system call instruction. A process’s thread can ‘‘block’’ in the kernel to wait for I/O, and resume where it left off when the I/O has finished.

    • p->state

      • the process is allocated/ready to run/running/waiting for IO/exiting

CHAPTER6

Overview

  • A file system shall
    • record files and directory tree
    • acknowledge vacancy spaces in disk
    • crash recovery e.x. power failuer
    • make sure file won’t be destroyed because of same time operation
    • mantain an in-memory cache of popular blocks
  • 7 layers
    • Disk
      • r/w blocks on an IDE hard drive
    • Buffer Cache
      • caches disk blocks and synchronizes access to them
    • Logging
      • allow higher layer to wrap updates to several blocks in a transaction.
      • ensures the blocks are updated atomatically in the face of crashes.
    • Inode
      • provides individual files
    • Directory
      • implement each dir
      • directory is a special kind of inode whose content is a sequence of dir entries.
    • Pathname
      • provides hierarchical path name s like /usr/rtm/xv6/fs.c
      • resolves them with recursive lookup
    • File Descriptor
      • abstracts many Unix resources (e.g. pipes devices files etx.)
      • using the file system interface simplifying the lives of app programmers
  • Blocks layout
    • Block 0 will never be used because it hold boot sector
    • Block 1 is called superblock, holds metadata about the file system
      • the file system size in blocks
      • the number of data blocks
      • the number of inodes
      • the number of blocks in the log
    • From Block 2 on
      • hold the log
      • hold inodes
      • bitmap
        • tracking which data blocks are in use
      • data blocks ~end

Buffer cache layer

  • Main Task
    • synchronize access to disk blocks to ensure that only one copy of a block is in memory and that only one kernel thread at a time uses that copy
    • cache popular blocks to increase accessing speed
  • kernel thread and cache
    • bread / bwrite(start) -> brelease(leave)

Logging layer

  • main task
    • prevent file system crash after system failure
  • problem solving
    • any system call can’t directly write the on-disk file system data structure.
    • system call write all it’s wishes on the log
    • after system call has logged all of its writes
  • after system failure
    • if log is marked as complete operation, then commit to system
    • otherwise, ignore
